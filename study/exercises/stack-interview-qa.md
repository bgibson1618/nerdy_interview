## TypeScript/JavaScript

> Prep note: This is your home turf — you review JS/TS by reflex, and it's exactly Nerdy's front-end and Node stack. So don't waste the interview re-explaining what a `Promise` is or why types help; the bar here is *senior depth and precision*. The traps in this language aren't the syntax, they're the mental models working devs carry around fuzzy: TS is a **compile-time-only, structurally-typed, fully-erased** layer bolted onto a **single-threaded, dynamically-typed runtime with a very specific event-loop ordering**. The two highest-value things to nail are (1) **types vanish at runtime, so trust boundaries need real runtime validation (zod)**, and (2) **microtask vs macrotask** ordering. Name the mechanism, not the definition.

### What you actually need to know (primer)

- **TypeScript is structural ("duck") typing, not nominal.** Two types are compatible if their *shapes* match, regardless of name or declared inheritance. `interface Dog { name: string }` and `interface Cat { name: string }` are mutually assignable, and a `{ name: string; age: number }` object satisfies a `Dog` parameter (excess properties are fine when passed through a variable; object *literals* get an extra "excess property" check). This is why "it compiled" doesn't mean "it's the type I meant." When you genuinely need nominal distinctions — a `UserId` that isn't interchangeable with an `OrderId`, both `string` underneath — you reach for **branded types**: `type UserId = string & { readonly __brand: 'UserId' }`. Contrast with Java/C#, where names define identity.

- **`unknown` vs `any` vs `never` — the type-safety hierarchy.** `any` *disables* the checker and is contagious — it silently spreads and defeats the whole point. `unknown` is the **type-safe top type**: it can hold anything, but you can't *use* it until you narrow (it forces a `typeof`/guard before access), which makes it the correct type for `JSON.parse`, `catch (e)`, and any untrusted input. `never` is the **bottom type** (no value inhabits it): it's the return type of a function that always throws or loops forever, the type of an impossible branch, and the secret to **exhaustiveness checking**. My rule of thumb: `any` is a code smell at a trust boundary — replace it with `unknown` + a guard.

- **Narrowing, discriminated unions, and exhaustiveness via `never`.** The compiler narrows on `typeof`, `instanceof`, `in`, truthiness, and equality. The senior pattern is a **discriminated (tagged) union** — a shared literal field (`kind`/`type`) that lets the compiler pick the member: `type Shape = { kind: 'circle'; r: number } | { kind: 'square'; side: number }`. In a `switch (s.kind)`, the `default` branch does `const _exhaustive: never = s` — if someone later adds a `'triangle'` variant and forgets to handle it, `s` is no longer assignable to `never` and **the build breaks**. That's compile-time enforcement of "did you handle every case," and a genuinely senior reflex.

- **Generics + constraints are about *preserving* type relationships, not just "any type."** `function identity<T>(x: T): T` keeps input and output linked; the constraint `<T extends { id: string }>` says "any `T`, as long as it has an `id`." The pattern people get fuzzy on is **`keyof` + indexed access**: `function get<T, K extends keyof T>(obj: T, key: K): T[K]` — the return type is *exactly* that property's type, so `get(user, 'age')` is `number`, not `string | number`. That's how libraries give you precise autocomplete instead of `any`.

- **`satisfies` is the operator most mid-level devs have never internalized.** A type annotation (`const config: Record<string, number> = {...}`) *widens* — you lose the specific keys. A cast (`as`) *lies* and turns off checking. `satisfies` does neither: it **validates that the value conforms to a type while preserving the narrow inferred type**. `const routes = { home: '/', login: '/login' } satisfies Record<string, string>` checks every value is a string *but* keeps `routes.home` as the literal `'/'` and the keys exact. Use it for config objects, palettes, and route maps where you want both the check *and* the precise type.

- **Utility types are derived types — learn the workhorses.** `Partial<T>` (all optional — patch/update payloads), `Required<T>` (all required), `Pick<T, K>` / `Omit<T, K>` (carve a subset — DTOs from a domain model), `Record<K, V>` (a map type), `Readonly<T>`, and the *inference* ones: `ReturnType<typeof fn>` and `Parameters<typeof fn>` (derive a type *from* a value so it can't drift). The principle: **derive types instead of duplicating them** — one source of truth, changes propagate. `ReturnType<typeof useUser>` beats re-typing the hook's shape by hand.

- **Declaration merging & module augmentation — the "how do I add a field to someone else's type" mechanism.** Interfaces with the same name merge; namespaces merge. The practical use is **module augmentation**: `declare module 'express' { interface Request { user?: User } }` to teach the type system that your auth middleware attached `req.user`. The same mechanism lets you extend `Window`, augment a third-party lib's types, or add a custom theme to a styling library — without forking. Knowing this separates "I use the types" from "I extend the type system."

- **Types are *fully erased* at runtime — the single most important TS fact.** `tsc` strips all type annotations; the emitted JS has *zero* knowledge of your interfaces. So `x instanceof MyInterface` is illegal (interfaces don't exist at runtime — only classes do), generics can't be reflected on, and — critically — **a value typed `User` coming off the wire is a *promise the compiler can't keep*.** `await res.json() as User` is a lie if the API sends garbage. At every trust boundary (HTTP responses, `localStorage`, env vars, webhook bodies, form input) you need **runtime validation** — `zod`/`io-ts`/`valibot` — which gives you a schema that both *checks* at runtime *and* infers the static type (`z.infer<typeof schema>`), so the static and runtime contracts can't diverge.

- **The event loop: microtasks vs macrotasks, and exact ordering.** JS is single-threaded with an event loop. **Macrotasks** (a.k.a. tasks): `setTimeout`, `setInterval`, I/O callbacks, message events. **Microtasks**: resolved-`Promise` `.then`/`await` continuations, `queueMicrotask`, `MutationObserver` callbacks. The rule that trips people up: **after each macrotask, the engine drains the *entire* microtask queue before rendering or running the next macrotask.** So `Promise.resolve().then(...)` always runs before a `setTimeout(..., 0)` queued at the same time. In Node, `process.nextTick` callbacks run ahead of even the Promise microtask queue. This is the classic "what logs in what order" gotcha (see Q2).

- **Closures, `this`, and the prototype chain (the runtime trio).** A **closure** captures *variables, not values* — a `var i` in a `for` loop is shared across callbacks and logs the final value (the historical interview bug; `let` fixes it by creating a fresh binding per iteration). **`this`** is determined by *how a function is called*, not where it's defined: method call → the object, `new` → the instance, `.call`/`.apply`/`.bind` → explicit, and a bare call → `undefined` in strict mode (the global object in non-strict). **Arrow functions have no own `this`** — they lexically capture the enclosing `this`, which is exactly why they fix the "lost `this` in a callback" problem (and why you should *never* use an arrow as an object method or class method you intend to rebind — `.bind` and `.call` can't change it). The **prototype chain** is JS's inheritance: property lookups walk the internal `[[Prototype]]` links (exposed as `__proto__`/`Object.getPrototypeOf`) until found or `null`; `class` is sugar over this.

- **The coercion / equality / identity footguns.** `==` does type coercion with genuinely surprising rules (`0 == ''` true, `null == undefined` true but `null == 0` false, `[] == ![]` true) — use `===` always; the only defensible `==` is `x == null` to catch both `null` and `undefined`. **Hoisting/TDZ**: `var` declarations hoist and read `undefined`; `let`/`const` hoist too but live in the **Temporal Dead Zone** — accessing before declaration throws `ReferenceError` (a *feature*, not a bug). **Pass-by-value-of-the-reference**: primitives copy by value; objects pass a *copy of the reference*, so mutating the object inside a function is visible outside, but reassigning the parameter isn't. And `structuredClone(obj)` matters because `{...obj}` and `Object.assign` are **shallow** — nested objects stay shared references, the source of "I copied it but mutating the copy changed the original" bugs.

- **ESM vs CommonJS, and the interop pain.** CommonJS (`require`/`module.exports`) is synchronous, dynamic, and was Node's original system. **ESM** (`import`/`export`) is the standard: statically analyzable (enables tree-shaking) and asynchronously loaded (so top-level `await` is allowed). The pain: they don't compose cleanly — `require()` of an ESM-only package historically threw (newer Node can require sync-friendly ESM, but it's still a sharp edge); default-export interop (`esModuleInterop`) is a perennial config headache; `__dirname` doesn't exist in ESM (you reconstruct it from `import.meta.url`); and `"type": "module"` in `package.json` flips a project's default. At a place like Nerdy this surfaces as build/bundler config friction and the "this package went ESM-only" upgrade tax.

- **`AbortController` is the standard cancellation primitive.** A `Promise` has no built-in cancel. `AbortController` gives a `signal` you pass to `fetch(url, { signal })` (or any abort-aware API); calling `controller.abort()` rejects the in-flight request with an `AbortError`. This is the correct way to cancel a stale request when a React component unmounts or a search input changes — the React data-fetching cleanup pattern. Knowing it (vs a leaked `isMounted` flag) is a senior tell.

- **Where this actually shows up at a company like Nerdy.** It's a learning marketplace (students, tutors, parents, staff) on **web + native mobile, AWS-first, many backend services** — so JS/TS is everywhere: React on the web, React Native / shared TS on mobile, Node services and Lambdas on the backend. The blind spots map directly to production incidents: **type erasure** → you *must* zod-validate webhook payloads (payments/CRM), API responses, and DynamoDB items, because a `User` type guarantees nothing at runtime; **floating promises / unhandled rejections** crash Node Lambdas; **event-loop blocking** (sync CPU work on the hot path) stalls every concurrent request on a single-threaded service; **`AbortController`** cancels stale tutor-search fetches on the front end; **shared-reference mutation** corrupts cached objects across requests; and **discriminated unions + exhaustiveness** model session states (`scheduled | live | ended | cancelled`) so a new state can't silently slip through unhandled.

### Graduated interview questions + model answers

**Q1 (basic) — "What does TypeScript actually give you over JavaScript, and what does it cost?"**
*Model answer / talking points:* "Static types catch a class of bugs at compile time, make large refactors safe across teams, and act as living contracts between modules — which is what matters at scale. But I'd go past the cliché: TS is *structural* and *compile-time only*. It's structurally typed, so compatibility is about shape, not name — 'it compiled' means the shapes lined up, not that I passed the object I intended. And every type is **erased** at compile time: the emitted JS has no idea your interfaces exist. So TS buys me developer-time safety and tooling, but at runtime I'm running plain JS with all of JS's dynamism — which is exactly why I still validate untrusted input at runtime." *(Naming structural + erased in the basic answer immediately signals depth.)*

**Q2 (mechanism) — "What does this log, and why?"** `console.log(1); setTimeout(() => console.log(2), 0); Promise.resolve().then(() => console.log(3)); console.log(4);`
*Model answer / talking points:* "`1, 4, 3, 2`. The synchronous code runs first — `1` then `4`. Then the engine drains the **microtask** queue before any macrotask, so the resolved-Promise `.then` (`3`) runs next. `setTimeout` is a **macrotask**, so `2` runs last, even with a `0`ms delay. The key mechanism: JS is single-threaded with an event loop, and after each macrotask it empties the *entire* microtask queue before moving on. Promise continuations and `await` are microtasks; timers and I/O are macrotasks. In Node, `process.nextTick` would jump ahead of even `3`." *(A flagship gotcha — getting the order *and* explaining the microtask-drain rule is high-value.)*

**Q3 (intermediate) — "`any` vs `unknown` vs `never` — when do you use each?"**
*Model answer / talking points:* "`any` turns the checker off and is contagious — I treat it as a smell, especially at a trust boundary. `unknown` is the type-safe top type: it holds anything but I can't *use* it without narrowing first, so it's the right type for `JSON.parse`, the `catch` binding, and untrusted input — it forces a guard. `never` is the bottom type, no value has it: it's what an always-throwing function returns and what an impossible branch collapses to. My favorite use of `never` is exhaustiveness — in a `switch` over a discriminated union, I assign the `default` case to a `never` variable, so if someone adds a union member and forgets to handle it, the build fails. So: `unknown` to safely *receive*, `never` to prove I handled *everything*, `any` almost never." *(The exhaustiveness trick is the senior flourish here.)*

**Q4 (advanced / GOTCHA) — "Your code is fully typed with `User` interfaces, the build is green, but in production a malformed API response causes a crash deep in your component. How did the type system let this through?"**
*Model answer / talking points:* "Because **types are erased at runtime** — they're a compile-time fiction. When I write `const user = await res.json() as User`, that `as` is an *assertion*, a promise I'm making to the compiler, not a check. If the API sends a different shape, nothing verifies it; the `User` type evaporates in the emitted JS, and I only find out when something downstream reads a field that's actually `undefined`. The fix is **runtime validation at every trust boundary** — HTTP responses, webhooks, `localStorage`, env vars, form input. I'd use **zod**: define a schema, `schema.parse(data)` throws on bad input *at the boundary* where I can handle it, and `z.infer<typeof schema>` gives me the static type from the same schema — so the runtime check and the compile-time type can't drift apart. The mental model: TS protects me from *my own* code; zod protects me from *the outside world*." *(The single highest-value answer to nail — it's the TS equivalent of GraphQL's N+1. It proves you understand the compile-time/runtime boundary, where most production TS incidents actually live.)*

**Q5 (tradeoffs) — "How do you cancel an in-flight request, and what's the wrong-but-common alternative?"**
*Model answer / talking points:* "Promises aren't cancellable on their own, so for fetches I pass an `AbortController`'s `signal` into `fetch` and call `abort()` when the work is stale — a component unmounts, or a search query changes and the old request is irrelevant. The wrong-but-common alternative is an `isMounted` boolean that just *ignores* the late response — that still lets the request complete (wasting bandwidth and a backend round trip) and is the source of race-condition bugs where an older, slower request resolves *after* a newer one and overwrites fresh data. `AbortController` actually cancels the request and rejects with an `AbortError` I can swallow. The tradeoff is a bit more wiring, but on a search-as-you-type surface — say tutor search — it's the difference between correct and a flaky stale-data bug." *(Naming the `isMounted` anti-pattern and the stale-response race is the senior signal.)*

**Q6 (judgment) — "Where do you draw the line on `==`, mutation, and shallow copies in a code review? What do you actually flag?"**
*Model answer / talking points:* "A few reflexes. I flag `==` and ask for `===` — coercion rules are surprising enough (`0 == ''`, `[] == ![]`) that the only `==` I'll accept is `x == null` as a deliberate null-and-undefined check. I flag mutation of shared objects, because JS passes a *copy of the reference* — mutating an argument is visible to the caller, and that's a classic bug when the object is cached or shared across requests in a Node service. And I flag `{...obj}` or `Object.assign` used as if they were deep copies: they're **shallow**, so nested objects stay shared — the 'I cloned it but editing the clone changed the original' trap. For a real deep copy I'd reach for `structuredClone`. I'd also watch for TDZ-adjacent issues and `var` where `let`/`const` belongs. None of these are nits — they're correctness, and they map straight to production incidents." *(Framing them as correctness, not style, is what separates a senior reviewer.)*

### One-liners to have ready
- "Types are erased at runtime — TS guards your code, zod guards the outside world."
- "TypeScript is structural, not nominal — compatibility is shape, not name."
- "Microtasks drain completely before the next macrotask — `.then` always beats `setTimeout(0)`."
- "`unknown` to safely receive, `never` to prove you handled every case, `any` almost never."
- "`satisfies` checks the type *and* keeps the narrow one — annotation widens, `as` lies."
- "Spread is a shallow copy; reach for `structuredClone` when nesting matters."
- "`this` is set by how a function is *called*; arrows capture it lexically."

## React

> Prep note: You ship React for a living, so this section skips the basics and goes straight for the depth a senior interviewer probes — the mental models you *use* but may not be able to *articulate* under pressure, and the gotchas you've half-internalized but can't always explain the *mechanism* for. The blind spots aren't "what is JSX"; they're render-vs-commit, why hooks depend on call order, why a stale closure eats your effect, when `useMemo` is theater, and what actually runs where in Server Components. Your job in the room is to name the *mechanism and the tradeoff*, not recite the API. Default to: "React re-renders by re-running the function — everything else follows from that."

### What you actually need to know (primer)

- **A render is React calling your function; "render" and "DOM update" are different phases.** React works in two phases. The **render phase** is pure and interruptible: React calls your component, builds a new element tree, and diffs it against the previous one. The **commit phase** is synchronous and side-effecting: React mutates the real DOM, then runs refs and `useLayoutEffect`, and finally — after the browser paints — flushes `useEffect`. The consequence: **re-rendering is cheap and frequent and does not mean the DOM changed.** If the diff finds nothing, the commit touches nothing. So "my component re-renders" is rarely the actual perf problem — the cost is a big reconcile or a real DOM mutation, not the function call. Because the render phase must be pure and re-runnable, side effects there (mutating a ref, firing analytics, `Math.random()` into JSX) are bugs — React's Strict Mode (18+) double-invokes renders in dev specifically to surface them.

- **Reconciliation is positional by default; keys override position — index keys are the trap.** React diffs the new tree against the old *by position in the children array* unless you give it a `key`. A `key` tells React "this is the *same* logical item even if it moved." Key a list by array index and the key is just the slot, so inserting at the top or reordering shifts every item's key — React thinks element 0 changed from "A" to "B" and *reuses the old DOM node and component state* for the wrong item. The visible bugs: a checked checkbox or focused input "follows the wrong row," or an animation replays. A stable key (a record's `id`) fixes it because identity now tracks the data, not the slot. Index keys are *only* safe for static, append-only, never-reordered lists. The deeper rule: **a component's state is tied to its position-in-the-tree plus its key** — change either and React unmounts/remounts, blowing away state.

- **The Rules of Hooks exist because hooks are stored in a positional list, not by name.** React doesn't know a hook is "the count state" — it stores hook state in a linked list on the fiber and advances through it on each `useState`/`useEffect`/etc. call. On the next render it must walk that list *in the same order* to hand each hook back its slot. So hooks must be called unconditionally, at the top level, in the same order every render — `if (x) useState()`, or a hook in a variable-length loop, desyncs the order and one hook gets another's state (React throws "rendered fewer/more hooks than during the previous render"). It's also why hooks only work inside components/other hooks: they need a fiber to attach to. Knowing *the list-and-order mechanism* — not just "don't put hooks in ifs" — is the senior tell.

- **Stale closures are the #1 hooks bug, and the deps array is how you opt out.** Every render is a snapshot: the function runs with that render's props and state captured in its closures. An effect, callback, or `setInterval` created in that render *sees that render's values forever* unless it's recreated. Classic bug: `useEffect(() => { const id = setInterval(() => setCount(count + 1), 1000); ... }, [])` — the interval closes over `count = 0` and just sets 1 every tick. Fixes: add `count` to deps (recreate the interval), use the **updater form** `setCount(c => c + 1)` so you never read the stale value, or stash the latest value in a ref. The deps array isn't a perf knob — it's **a correctness contract: it declares everything the effect reads so React can re-run it when those change.** Lying about deps (omitting one to "stop it re-running") ships stale-data bugs; the real fix is the updater form, a ref, or `useReducer`.

- **`useMemo`/`useCallback` are about referential equality, not raw speed — most uses are cargo-culted.** They cache a value/function *by reference* across renders while deps don't change. They matter in two cases: (1) the memoized thing is passed to a `React.memo` child or used in another hook's deps, where a *fresh reference every render* would defeat the bailout or needlessly re-fire the effect; or (2) the computation is genuinely expensive (a big sort/filter). Wrapping a cheap `() => x + 1` in `useCallback` *adds* work — the deps comparison and closure allocation cost more than they save, and the code is harder to read for nothing. The honest framing: **memoization is a referential-identity tool first, a CPU tool second.** And it's load-bearing only if the *whole chain* is memoized — a `React.memo` child still re-renders if even one prop is a fresh object/array literal. React 19's compiler auto-inserts these, which is the strongest signal that hand-writing them was always mechanical.

- **React 18 batches state updates by default; effects run after paint, `useLayoutEffect` before it.** Pre-18, React only batched updates inside its own event handlers; updates in promises, `setTimeout`, or native callbacks each triggered a separate render. **Automatic batching** in 18 batches them everywhere, so multiple `setState` calls in an async callback now produce one render (use `flushSync` to opt out when you truly need a synchronous DOM update mid-handler). On timing: **`useEffect` runs asynchronously *after* paint**, so it won't block visual updates — good for data fetching, subscriptions, logging. **`useLayoutEffect` runs synchronously *after* DOM mutation but *before* paint**, so use it (sparingly) when you must read layout and mutate before the user sees a flicker — positioning a tooltip, say. Overusing it blocks paint and hurts perceived perf.

- **Controlled vs uncontrolled is about who owns the value — and Context broadcasts to its whole subtree.** A **controlled input** has its `value` driven by React state (`value={x} onChange={...}`); React is the source of truth, which you need for validation, formatting, or cross-field logic, at the cost of a render per keystroke. An **uncontrolled input** lets the DOM hold the value and you read it via a ref (or `defaultValue`) on submit — less code, fewer renders, but you can't easily react to changes mid-edit. Separately: **`useContext` re-renders every consumer in the subtree whenever the Provider's `value` changes by `Object.is`** — and an inline `value={{user, setUser}}` is a *new object every render*, so every consumer re-renders whenever the provider's parent renders. The senior gotcha: **`React.memo` does *not* stop a context-driven re-render** — a consumer re-renders on context change regardless of its props. Fixes: memoize the value, **split contexts** (a rarely-changing `AuthContext` apart from a chatty `ThemeContext`), or push state down so fewer components subscribe. Context isn't a state manager and isn't free — it's an implicit prop drill with a broadcast cost.

- **Refs, `forwardRef`, `useImperativeHandle`, and the "remount with key" trick are the escape hatches.** A **ref** is mutable, persists across renders, and *does not trigger a re-render when you mutate it* — perfect for DOM nodes, interval IDs, "previous value," or any latest-value-without-rerender pattern. `forwardRef` lets a parent's ref reach a child's DOM node (in React 19, `ref` is a normal prop, so `forwardRef` is no longer needed); `useImperativeHandle` exposes a *curated* imperative API (`{ focus, scrollToTop }`) instead of the raw node — useful for design-system components. And the highest-leverage trick: **changing a component's `key` forces React to unmount the old instance and mount a fresh one**, resetting all its state — the idiomatic way to "reset this form when the selected user changes" (`<Form key={userId} />`) instead of wiring an effect to clear fields.

- **Concurrent features and the Server/Client split are the modern depth.** **`useTransition`** marks a state update non-urgent so React can interrupt it to keep the UI responsive (the input stays snappy while an expensive filtered list lags behind); **`useDeferredValue`** does the same by deferring a *value* rather than wrapping the setter. **`Suspense`** lets a component "suspend" while data/code loads and shows a fallback; with a Suspense-aware data layer it replaces a lot of manual `isLoading` flags. **Error boundaries** catch errors thrown *during render* in their subtree and show a fallback — and the gotcha: **they must be class components** (via `getDerivedStateFromError`/`componentDidCatch`; there's no hook equivalent), and they **can't catch errors in event handlers, async code, SSR, or their own render** — you handle those with try/catch. **Server Components (RSC)** run *only* on the server, never ship their JS to the browser, and can directly `await` a DB/service call; **Client Components** (the file is marked `'use client'`) render on the server too (SSR/prerender) but also run and hydrate in the browser, and are the only ones that can use state, effects, refs, or browser APIs. `'use client'` is a *boundary*, not a per-component tag — everything imported into a client module becomes client code. **Where this shows up at a company like Nerdy:** a learning marketplace with web + native means React for web and likely React Native for the apps; a tutor-search page benefits from `useTransition`/`useDeferredValue` so typing stays smooth while results recompute; a live-classroom UI leans on controlled inputs, refs for media/whiteboard nodes, and `useLayoutEffect` for measuring/positioning; on the Next.js App Router, RSCs render student dashboards on the server (fetching from backend services directly) and ship `'use client'` islands only for the interactive bits — cutting bundle size for students on mobile networks; and context-splitting/memoization matter because auth/user context is read by nearly every component, so a sloppy provider re-renders the whole app on every keystroke.

### Graduated interview questions + model answers

**Q1 (basic) — "What actually happens when a component 're-renders'?"**
*Model answer / talking points:* "React calls my component again, which produces a new element tree, and React diffs it against the previous one — that's the *render phase*, and it's pure and cheap. It then *commits* only the differences to the real DOM in a separate phase. The thing people miss is that re-rendering doesn't mean the DOM changed: if the diff finds nothing different, nothing gets mutated. So a re-render is mostly the cost of running the function and reconciling — it's only expensive when the tree is large or there's an actual DOM mutation. That's why I don't reflexively optimize re-renders; I profile to see whether the reconcile or commit is the real bottleneck."

**Q2 (mechanism) — "Why can't you call a hook inside an `if` or a loop? What's the actual mechanism?"**
*Model answer / talking points:* "Because React identifies hooks by *call order*, not by name. It keeps each component's hook state in a list on the fiber and walks it in order on every render — first `useState` gets the first slot, second gets the next, and so on. On the next render it has to call them in the same order to hand each one back its slot. If I conditionally skip a hook, every subsequent hook shifts by one and gets the wrong state — React detects the count mismatch and throws 'rendered fewer hooks than expected.' So 'call hooks unconditionally at the top level' is really 'keep the positional order stable.' If I need conditional logic, I put the condition *inside* the hook, or split the branch into its own always-rendered component."

**Q3 (intermediate) — "Why do keys matter in lists, and what specifically goes wrong with index keys?"**
*Model answer / talking points:* "Keys are how React tracks element *identity* across renders. Without a key, React diffs children by position; with a key, it matches old and new elements by key even if they moved. If I use the array index as the key, the key is just the position — so the moment I insert at the top or reorder, every item's key shifts, and React thinks item 0's data changed rather than that a new item appeared. It then *reuses the existing DOM node and component state* for the wrong row. Classic symptoms: a focused input or checked checkbox sticking to the wrong row after a reorder, or an animation replaying. The fix is a stable key from the data itself, like a record id. Index keys are only fine for static, append-only lists that never reorder."

**Q4 (advanced / gotcha) — "Walk me through a stale-closure bug in a `useEffect`. Why does it happen and how do you fix it?"**
*Model answer / talking points:* "Every render is a snapshot — the function runs with that render's props and state baked into its closures. So if I write `useEffect(() => { const id = setInterval(() => setCount(count + 1), 1000); return () => clearInterval(id); }, [])`, the interval closes over `count` from the *first* render, which is `0`, forever. Every tick computes `0 + 1`, so it sets 1 and stops climbing. The mechanism is that the empty deps array told React never to recreate the effect, so it never captures a newer `count`. Three honest fixes: add `count` to the deps so the interval is recreated when it changes; better, use the updater form `setCount(c => c + 1)` so I never read the stale value at all; or stash the latest value in a ref and read `ref.current` inside the interval. The meta-point is that the deps array is a *correctness* contract, not a perf knob — it lists what the effect reads. The anti-pattern is omitting a dep to silence the lint warning, which just hides the staleness. I reach for the updater form or `useReducer` first." This is the single highest-value answer to nail — it's React's equivalent of the GraphQL N+1: the failure mode that proves you understand the model, not just the API.

**Q5 (tradeoffs) — "When does `useMemo`/`useCallback` actually help, and when is it just noise?"**
*Model answer / talking points:* "They cache a value or function *by reference* across renders, so they help in exactly two cases: when the result feeds a `React.memo` child or another hook's dependency array — where a fresh reference every render would break the memo bailout or needlessly re-fire the effect — or when the computation itself is genuinely expensive, like a large sort or filter. For a cheap inline function or `x + 1`, wrapping it is *net negative*: you pay for the deps comparison and the closure allocation and add noise. And memoization is only load-bearing if the whole chain holds — a `React.memo` child still re-renders if even one prop is a fresh object literal, so memoizing the callback but passing `style={{...}}` inline defeats it. I treat them as referential-identity tools first and CPU tools second. Worth noting React 19's compiler auto-inserts most of these, which tells you hand-writing them was always mechanical."

**Q6 (judgment) — "We notice the whole app re-renders on every keystroke in a form. The form uses a global Context for user/auth state. What's likely wrong and how do you fix it?"**
*Model answer / talking points:* "My first suspect is the Context provider. `useContext` re-renders *every* consumer in the subtree whenever the provider's `value` changes — and if the value is an inline object like `value={{user, setUser, ...}}`, that's a brand-new object on every render of the provider's parent, so every consumer re-renders even when nothing meaningful changed. And `React.memo` won't save them — context change bypasses it. If the form's keystroke state lives high enough to re-render that provider, the whole tree goes with it. The fixes, in order: memoize the context value so its reference is stable; **split the context** so rarely-changing auth state isn't in the same provider as chatty state; and *colocate* the form's input state down in the form component so a keystroke only re-renders the form. I'd confirm with the Profiler's 'why did this render' before changing anything. The principle is lifting state only as high as it needs to go, and treating Context as a broadcast with a real cost, not a free global store."

### One-liners to have ready
- "A render is just React calling your function — it doesn't mean the DOM changed."
- "Hooks are matched by call *order*, not name — that's the whole reason for the rules."
- "Index keys are fine until the list reorders; then state follows the wrong row."
- "The deps array is a correctness contract, not a performance knob — lying about it ships stale-data bugs."
- "`useMemo`/`useCallback` are about referential equality first, CPU second — most uses are cargo cult."

## MySQL

> Prep note: You have real hands-on SQL/MySQL experience, so this section skips the relational-vs-NoSQL 101 and goes at the depth a senior interviewer probes for — the parts experienced devs are *fuzzy* on rather than ignorant of. The blind spots that separate "I write queries" from "I understand the engine": how InnoDB physically stores rows (the clustered index *is* the primary key), why a secondary-index hit can mean a second lookup, the leftmost-prefix rule for composite indexes, why MySQL defaults to REPEATABLE READ when nearly everyone else defaults to READ COMMITTED, how MVCC lets readers not block writers, and how gap/next-key locks quietly stop phantoms (and quietly cause deadlocks). Confident senior register: name the mechanism and the tradeoff, not the definition. "EXPLAIN says it's using an index" is junior; "EXPLAIN shows `ref` on a covering index so it never touches the clustered leaf" is the bar.

### What you actually need to know (primer)

- **A B-tree (really a B+tree) index is a sorted, balanced, high-fan-out tree — that's *why* it's fast and *why* it has rules.** Keys are stored in sorted order and each node holds many of them, so a table with millions of rows is only ~3–4 levels deep and a point lookup is a handful of page reads (`O(log n)`). The sorted structure is what makes it good for equality (`=`), ranges (`>`, `<`, `BETWEEN`), prefix `LIKE 'abc%'`, and `ORDER BY` on the indexed columns — and useless for anything that can't exploit the ordering: leading-wildcard `LIKE '%abc'`, a function on the column (`WHERE YEAR(created_at) = 2024` can't use an index on `created_at` unless it's a functional index), or `!=`. The "balanced" part means it stays shallow under inserts via node splits — which is also the write cost of an index.

- **InnoDB tables ARE a clustered index, and the cluster key is the PRIMARY KEY.** This is the single biggest InnoDB mental model. The rows aren't a heap pointed at by indexes (that's MyISAM / SQL Server's default / Postgres heaps) — the *entire row lives in the leaf nodes of a B+tree ordered by the primary key*. The consequences cascade: (1) PK lookups are maximally fast — you reach the row at the leaf, no second hop; (2) inserts in PK order are cheap, while random PK inserts (e.g. a random UUIDv4 PK) cause page splits and fragmentation — which is exactly why `AUTO_INCREMENT` (or a time-ordered ULID/UUIDv7) PK is preferred; (3) a huge or changing PK bloats *every* secondary index. If you don't declare a PK, InnoDB uses the first unique non-null index, or invents a hidden 6-byte `row_id` — so "no primary key" is never really true.

- **Secondary indexes store the PK, not a row pointer — so they often cost a *second* lookup.** A secondary index's leaf doesn't hold a physical row address; it holds the **primary key value** of the matching row. So `WHERE email = ?` on an index `(email)` finds the PK in the secondary index, then does a **second B+tree traversal of the clustered index** to fetch the rest of the row. That second hop is the "bookmark lookup." It's also why a fat PK is doubly expensive: it's copied into every secondary index's leaves.

- **A covering index makes the second lookup disappear.** If every column a query needs is already *in the index* (the indexed columns plus the PK that rides along for free), InnoDB answers entirely from the secondary index and never touches the clustered index. EXPLAIN shows this as **`Using index`** in the Extra column. Example: index `(user_id, status)` covers `SELECT status FROM sessions WHERE user_id = ?` — no row fetch. This is the mechanical reason `SELECT *` is dangerous: it pulls columns not in your index, forcing the row lookup and defeating an otherwise-covering plan. "Don't `SELECT *`" isn't style — it's "don't break covering indexes and don't ship columns over the wire you never read."

- **The leftmost-prefix rule governs composite indexes — and it's the most-missed index fact.** An index on `(a, b, c)` is a *single structure sorted by a, then b, then c*. It serves `WHERE a`, `WHERE a AND b`, and `WHERE a AND b AND c` — and `ORDER BY a, b, c`. It **cannot** serve `WHERE b` alone, `WHERE c` alone, or `WHERE b AND c`, because the data isn't sorted by `b` until you've fixed `a`. Subtle corollary: a **range** on a column stops the prefix — `WHERE a = ? AND b > ? AND c = ?` uses the index for `a` and the `b` range, but `c` can't be an index seek (rows aren't sorted by `c` across different `b` values). So column *order* is a design decision: equality columns first, then the range/sort column. It's also why you don't need a separate single-column index on `a` if you already have `(a, b, c)`.

- **Selectivity / cardinality decides whether an index is even worth using.** Selectivity = distinct values ÷ total rows. An index on a low-cardinality column like `gender` or `is_active` (2–3 values) is nearly useless: matching half the table through an index means thousands of random secondary→clustered lookups, which is *slower* than scanning sequentially — so the optimizer will (correctly) ignore it and do a full scan. High-cardinality columns (email, user_id) are where indexes pay off. Practical lesson: indexing a boolean is usually wasted write cost; if you must, it's only useful as the *trailing* part of a composite led by a selective column.

- **ACID, isolation levels, and the anomaly each prevents — plus MySQL's surprising default.** The anomalies, in order of nastiness: **dirty read** (reading another transaction's *uncommitted* changes), **non-repeatable read** (you read a row twice in one transaction and get different values because someone committed an `UPDATE` between), and **phantom read** (you run the same `WHERE` twice and a *new row* appears because someone committed an `INSERT`). The levels: `READ UNCOMMITTED` (allows all), `READ COMMITTED` (stops dirty reads), `REPEATABLE READ` (stops dirty + non-repeatable), `SERIALIZABLE` (stops all). **The gotcha senior devs get wrong: InnoDB defaults to `REPEATABLE READ`, while Postgres, Oracle, and SQL Server default to `READ COMMITTED`.** More surprising still — InnoDB's REPEATABLE READ *also blocks phantoms* (the SQL standard doesn't require this): plain reads avoid them via the MVCC snapshot, and *locking* reads/writes use **next-key locks** (a row lock plus a gap lock on the range before it). So the same code can behave differently across engines, and "it worked on Postgres" is a real migration trap.

- **MVCC is why readers don't block writers (and vice versa).** InnoDB keeps **multiple versions** of each row. A transaction reads a **consistent snapshot** — established at its *first consistent read* in REPEATABLE READ, or fresh per statement in READ COMMITTED — reconstructed from the **undo log**. A plain `SELECT` (a non-locking "consistent read") never takes row locks and never waits on writers; it just reads the version visible to its snapshot. Writers create new versions, and old versions stay in the undo log for any open snapshot that still needs them. The cost: a long-running transaction holds back **undo-log purge**, bloating the undo log / history list and slowing everyone — the classic "that analytics transaction left open for an hour is killing the DB" incident. Note that `SELECT ... FOR UPDATE` / `FOR SHARE` opt *out* of MVCC into actual locking reads.

- **Row locks vs table locks, and how deadlocks actually happen.** InnoDB does **row-level locking** (it locks index records), which is why it scales for concurrent OLTP — far better than table-level locking. But because locks are taken on **index records**, a query that can't use an index may lock far more rows than you expect (it effectively scans and locks). A **deadlock** is two transactions each holding a lock the other needs (T1 locks A then wants B; T2 locks B then wants A); InnoDB **detects the cycle and rolls back one transaction** (`ERROR 1213`), so the app must **retry** the loser. Prevention is **consistent lock ordering**, short transactions, and avoiding gratuitously wide `WHERE` clauses in updates. Gap/next-key locks make deadlocks *more* likely under REPEATABLE READ than READ COMMITTED — a real reason teams sometimes switch.

- **EXPLAIN: read the `type` column first; top to bottom is best to worst.** `system`/`const` (PK/unique lookup, one row — ideal), `eq_ref` (unique-key join), `ref` (non-unique index match — good), `range` (index range scan — fine), `index` (full *index* scan — reads the whole index, better than the table but still everything), `ALL` (**full table scan — the red flag**). Then read **`key`** (which index was actually chosen), **`rows`** (estimated rows examined), and **`Extra`**: `Using index` = covering (great), `Using where` = filtering after fetch, `Using filesort` = sorting in memory/disk because no index satisfied `ORDER BY`, `Using temporary` = a temp table (often `GROUP BY`/`DISTINCT` with no usable index). `EXPLAIN ANALYZE` actually *runs* the query and shows real vs. estimated rows, which exposes bad optimizer estimates — usually stale statistics, fixed with `ANALYZE TABLE`.

- **Normalization vs deliberate denormalization is a judgment call, not a religion.** Normalize (3NF) by default — one fact in one place — to avoid update anomalies and keep writes cheap and consistent. **Denormalize deliberately** for read-heavy hot paths where the join or fan-out hurts: a cached `session_count` on a user row, a duplicated `tutor_name` on a session so a list view skips a join. The tradeoff to name: denormalization buys read speed at the cost of **write complexity and consistency risk** — you now have to keep the copies in sync via triggers, application logic, or async jobs, each a place for drift. Senior framing: "I normalize for correctness and denormalize for measured read pressure, with a clear owner of the derived data."

- **Replication, replica lag, and connection pooling — the operational layer.** MySQL replication is typically **asynchronous**: writes go to the **primary**, which streams its **binlog** to **replicas** that replay it. You scale *reads* by routing `SELECT`s to **read replicas**, but async replication means **replica lag** — a replica can be milliseconds to seconds behind, so **read-your-own-writes** breaks (user updates their profile, the read hits a lagging replica, shows stale data). Fixes: route reads-after-writes to the primary, use semi-synchronous replication, or tolerate the staleness. **Connection pooling** matters because every MySQL connection is a real backend thread with non-trivial memory; opening one per request (common in serverless/Lambda) exhausts `max_connections` and adds handshake latency — a pool (app-side, or a proxy like **RDS Proxy** / **ProxySQL**) reuses warm connections. **`AUTO_INCREMENT`** gives monotonic surrogate PKs (good for clustered-index insert locality), but ids can have gaps (rolled-back inserts consume values) and are per-table, not globally unique.

- **Where this actually shows up at a company like Nerdy.** Nerdy (Varsity Tutors) is an AWS-first learning marketplace — students, tutors, parents, staff — with web + native mobile and many backend services, so MySQL almost certainly runs on **RDS or Aurora MySQL**. The patterns that bite: a **session-list or dashboard query** that's secretly N+1 from the app layer, or a slow `WHERE student_id = ? AND status = ?` that needs a composite `(student_id, status)` index (leftmost-prefix!); **read replicas** serving heavy dashboard/reporting reads while writes go to the primary, with **replica lag** causing "I just booked a session but don't see it"; **Lambda/serverless** functions exhausting `max_connections` without **RDS Proxy** pooling; **deadlocks** under concurrent booking/scheduling writes that need retry logic; and **denormalized** counters (unread messages, upcoming sessions) on hot profile rows. Aurora specifically changes the replication story — it shares one distributed storage layer, so replica lag is typically single-digit milliseconds rather than the seconds you can see with classic binlog replication.

### Graduated interview questions + model answers

**Q1 (basic) — "What's an index, and what's the cost of adding one?"**
*Model answer / talking points:* "An index is a separate sorted B+tree on one or more columns that lets the engine find rows in roughly `O(log n)` page reads instead of scanning the whole table. It speeds up reads on the indexed columns — equality, ranges, prefix matches, `ORDER BY`. The cost is on the **write** side: every `INSERT`/`UPDATE`/`DELETE` has to maintain every affected index, plus storage. So I index for the **actual query patterns** in `WHERE`/`JOIN`/`ORDER BY`, not every column — over-indexing slows writes and wastes space. The decision is always read benefit vs. write/storage cost."

**Q2 (mechanism) — "In InnoDB, what's the difference between the primary-key index and a secondary index, and why does it matter for performance?"**
*Model answer / talking points:* "In InnoDB the table **is** a clustered index — the full row lives in the leaf nodes of a B+tree ordered by the primary key, so a PK lookup lands you directly on the row, no second hop. A **secondary index** is a separate B+tree whose leaves store the indexed column(s) plus the **primary key value** — *not* a physical row pointer. So a secondary-index lookup finds the PK, then does a **second traversal of the clustered index** to fetch the rest of the row — the bookmark-lookup cost. Two big implications: a **fat primary key** bloats every secondary index because it's copied into all their leaves, and a **covering index** — one that already holds every column the query needs — skips the second lookup entirely. That's also why a random-UUID PK hurts insert locality and an `AUTO_INCREMENT` PK helps it."

**Q3 (intermediate) — "I have an index on `(student_id, session_date, status)`. Which of these can use it: filter by `student_id`; by `session_date` alone; by `student_id` and `session_date`; by `status` alone?"**
*Model answer / talking points:* "This is the **leftmost-prefix rule**. The index is sorted by `student_id`, then `session_date`, then `status`. So: **`student_id` alone — yes**. **`student_id` + `session_date` — yes**. **`session_date` alone — no**, because the data isn't ordered by `session_date` until `student_id` is fixed. **`status` alone — no**, same reason. The mental model is one structure sorted left to right — you can use a contiguous prefix from the left, but you can't skip a column. One subtlety: a **range** on a middle column stops the prefix — `student_id = ? AND session_date > ?` uses the index up to and including the `session_date` range, but `status` after a range can't be a seek. So column order matters: equality-filter columns first, range/sort column last."

**Q4 (advanced / gotcha) — "Your `EXPLAIN` shows `type: ALL`. Walk me through diagnosing and fixing it — and what would `Using filesort` tell you?"**
*Model answer / talking points:* **This is the single highest-value answer to nail — it's the MySQL equivalent of GraphQL's N+1; reading a plan is what separates "writes queries" from "understands the engine."** "`type: ALL` means a **full table scan** — no usable index, so it's reading every row. First I check the `WHERE`/`JOIN`/`ORDER BY` columns and whether an index exists *and is usable* — a common trap is the index existing but being unusable because of a **function on the column** (`WHERE DATE(created_at) = ?`), an **implicit type cast** (a string compared to a numeric column), or a **leading wildcard** `LIKE '%foo'`. If there's genuinely no index, I add a composite one matching the filter, ideally **covering** so EXPLAIN flips to `Using index`. I also sanity-check **selectivity** — if the predicate matches most of the table, a scan is actually correct and an index won't help. **`Using filesort`** means MySQL had to sort the result *outside* index order — it couldn't satisfy `ORDER BY` from the index, so it sorts in a buffer (spilling to disk on big sets). The fix is an index whose column order matches the `ORDER BY` (with any equality filters before it), so rows come back pre-sorted. `Using temporary` similarly flags a temp table, often from `GROUP BY`/`DISTINCT` without a supporting index."

**Q5 (tradeoffs) — "InnoDB defaults to REPEATABLE READ. Walk me through the isolation levels, what each prevents, and whether you'd ever change the default."**
*Model answer / talking points:* "Four anomalies, increasing severity: **dirty read** (seeing uncommitted data), **non-repeatable read** (a row's value changes between two reads in your transaction because someone committed an update), and **phantom read** (a new row appears for the same `WHERE` because someone committed an insert). The levels stop progressively more: `READ UNCOMMITTED` (nothing), `READ COMMITTED` (dirty reads), `REPEATABLE READ` (dirty + non-repeatable), `SERIALIZABLE` (everything — InnoDB does it by implicitly turning plain `SELECT`s into locking `FOR SHARE` reads). The thing people miss: **InnoDB defaults to REPEATABLE READ, whereas Postgres, Oracle, and SQL Server default to READ COMMITTED** — so identical code can behave differently across engines, a real migration gotcha. And InnoDB's REPEATABLE READ also blocks **phantoms** — plain reads via the MVCC snapshot, locking reads/writes via **next-key (gap) locks** — which the SQL standard doesn't require. Would I change it? Sometimes to **READ COMMITTED**: high-concurrency OLTP can hit fewer gap-lock deadlocks and hold locks for shorter durations, at the cost of the stable snapshot. It's a deliberate tradeoff — snapshot stability vs. concurrency."

**Q6 (judgment) — "How do you reason about read replicas and replica lag, and when does denormalization make sense?"**
*Model answer / talking points:* "Replication in MySQL is usually **async**: writes hit the **primary**, which ships its binlog to **replicas** that replay it. I scale reads by routing `SELECT`s to replicas — but async means **replica lag**, so the trap is **read-your-own-writes**: a user updates something, the immediate read hits a lagging replica, and they see stale data. I handle it by routing reads right after a write to the primary, using semi-sync or Aurora (single-digit-ms lag thanks to shared storage), or tolerating staleness where it's harmless. On **denormalization**: I normalize by default for write correctness and denormalize **deliberately** only for measured read pressure — a cached counter or a duplicated display name to skip a join on a hot list view. The cost I always name is **consistency**: I now own keeping the copies in sync, via app logic, triggers, or async jobs, each a place for drift. So the judgment is: normalize for correctness, denormalize for proven read hotspots with a clear owner of the derived data — never speculatively."

## SQL Server

> Prep note: You have real hands-on SQL Server experience, so this section skips the beginner basics and goes straight for senior-level depth and the differences-from-MySQL that bite. Assume you know what an index and a transaction are; the value here is precise mental models for the things experienced devs stay fuzzy on — how SQL Server's clustered index *is* the table, why the default isolation behaves differently than MySQL's, and the parameter-sniffing trap behind "the proc was fast yesterday." In the interview, name the *mechanism* and the *tradeoff*, not the definition. The single highest-value thing you can demonstrate is reading an actual execution plan and reasoning about *why* the optimizer chose a scan over a seek — that's the line between "wrote some T-SQL" and "owns the database."

### What you actually need to know (primer)

- **The clustered index IS the table — and you get exactly one.** A clustered index physically orders the rows by its key; the table's data pages *are* the leaf level of that index, so "the clustered index" and "the row storage" are the same structure. One per table — a table without one is a *heap*. The gotcha vs MySQL/InnoDB: InnoDB silently clusters on the PRIMARY KEY for you, but SQL Server lets you make the PK *nonclustered* and cluster on something else. By default `PRIMARY KEY` creates the clustered index *if the table doesn't already have one*, but that's a default you can and sometimes should override — e.g. cluster on a narrow, monotonically increasing key to avoid page splits even when the logical PK is a GUID. Pick the wrong clustered key (a random `NEWID()` GUID) and every insert lands mid-table, fragmenting it and splitting pages.

- **Nonclustered indexes are separate B-trees that point back to the row — and INCLUDE makes them "covering."** A nonclustered index stores its key columns plus a *row locator* (the clustered key, or a RID for a heap). When a query needs columns the index doesn't have, the engine does a **key lookup** back into the clustered index — one extra read *per row*, which silently turns a fast seek into a slow plan under volume. The fix is a **covering index** via `INCLUDE`: `CREATE INDEX IX_x ON Sessions(TutorId) INCLUDE (StartsAt, Status)` keeps `StartsAt`/`Status` in the leaf pages *without* making them part of the sortable key — so the lookup disappears, and they don't bloat the B-tree's intermediate levels the way real key columns would. Knowing the difference between a key column (sortable/seekable) and an INCLUDE'd column (just-along-for-the-ride payload) is a strong senior signal.

- **The default isolation is READ COMMITTED with *locking* — not MVCC — and that surprises people coming from MySQL.** Out of the box, SQL Server's READ COMMITTED takes shared locks for reads, so readers can block writers and vice versa. MySQL/InnoDB defaults to **REPEATABLE READ** backed by MVCC, so two engines running "the same" SQL behave differently around blocking and phantoms. SQL Server's optimistic, MVCC-style answer is opt-in: **SNAPSHOT** isolation and **READ_COMMITTED_SNAPSHOT (RCSI)**. RCSI is the high-leverage one — flip a database-level setting and your existing READ COMMITTED queries start reading a *row version* instead of taking shared locks, so readers stop blocking writers with no query changes. The cost: the **version store lives in tempdb**, so you trade lock contention for tempdb I/O and growth. "Readers don't block writers anymore, paid for in tempdb" is the one-sentence model.

- **SNAPSHOT vs RCSI is a real distinction, not a synonym.** Both use row-versioning, but **SNAPSHOT** gives *transaction-level* consistency — the whole transaction sees a frozen point-in-time view (as of the transaction's start) and can hit **update conflicts (error 3960)** if another transaction changed a row it read. **RCSI** gives only *statement-level* consistency — each statement sees data as of *its own* start, and there are no update-conflict errors. RCSI is the drop-in fix for blocking; SNAPSHOT is for when you genuinely need a stable view across multiple statements. Conflating them in an interview is a tell.

- **T-SQL dialect specifics that differ from MySQL — have these reflexes.** `TOP (n)` instead of `LIMIT n`; standardized paging is `ORDER BY ... OFFSET @n ROWS FETCH NEXT @m ROWS ONLY` (OFFSET *requires* an ORDER BY). `IDENTITY(1,1)` instead of `AUTO_INCREMENT`; `SCOPE_IDENTITY()` to read the value back safely — *not* `@@IDENTITY`, which can return an identity generated by a trigger on another table. **`NVARCHAR`** is Unicode (UTF-16, 2 bytes per code unit) vs `VARCHAR` (single-byte by codepage — though with a `_UTF8` collation on 2019+ `VARCHAR` can store UTF-8); Unicode string literals need the `N'...'` prefix, and a non-`N` literal silently round-trips through the column's codepage. Objects live in **schemas** (`dbo.Sessions`) — two-part-name your references. `GETDATE()`/`SYSUTCDATETIME()`, `ISNULL()` vs `COALESCE()`, `LEN()` vs `DATALENGTH()`. `MERGE` exists for upserts but has enough documented bugs and concurrency footguns that many seniors avoid it in favor of explicit `UPDATE`-then-`INSERT`. Error handling is `BEGIN TRY / BEGIN CATCH` with `THROW` (modern) or `RAISERROR` (legacy). **Table variables (`@t`) vs temp tables (`#t`)**: temp tables get statistics so the optimizer estimates them realistically; table variables historically estimate **1 row** regardless of contents (mitigated by table-variable deferred compilation on 2019+ and by `OPTION (RECOMPILE)`), which can produce catastrophic plans on large sets — reach for `#temp` when the set is big.

- **Stored procedures cache execution plans — and that's where PARAMETER SNIFFING bites.** This is the SQL Server equivalent of GraphQL's N+1: the *classic* gotcha you must know cold. On first execution the optimizer "sniffs" the parameter values passed and builds a plan optimized *for those specific values*, then caches it. If that compiling call passes an atypical value (a tutor with 3 sessions), the cached plan — say a nested-loop seek — gets reused for a later call with a wildly different value (a tutor with 3 million sessions), where a hash join and scan would've been right. So "the proc was fast for weeks and suddenly crawled" usually means a bad plan got cached after a recompile, stats update, or restart. Fixes, in order of bluntness: `OPTION (RECOMPILE)` (recompile every execution — fresh plan, costs CPU), `OPTIMIZE FOR (@p = <value>)` or `OPTIMIZE FOR UNKNOWN` (plan for a representative/average value instead of the sniffed one), or splitting into separate procs/branches. SQL Server 2022 added **Parameter Sensitive Plan (PSP) optimization** to cache multiple plans per statement, but you should still understand the underlying mechanism.

- **Window functions and the optimizer are the senior-depth flex.** Window functions (`ROW_NUMBER()`, `RANK()`, `DENSE_RANK()`, `LAG`/`LEAD`, running totals via `SUM() OVER (PARTITION BY ... ORDER BY ... ROWS BETWEEN ...)`) do per-group analytics without self-joins — the canonical "latest row per group" / dedupe pattern is `ROW_NUMBER() OVER (PARTITION BY TutorId ORDER BY StartsAt DESC)`, then filter `= 1`. On reading plans: SQL Server's optimizer is **cost-based**, driven by **statistics** (histograms of column value distribution). Internalize **seek vs scan** (a seek navigates the B-tree to specific rows; a scan reads the whole index/table), **key lookups** (the covering-index smell above), and the gap between **estimated and actual row counts** — a big gap on the actual plan means *stale or missing statistics* and is the first thing to suspect when a plan goes wrong. `SET STATISTICS IO ON` to see logical reads is often more honest than wall-clock time.

- **Where this actually shows up at a company like Nerdy.** Nerdy (Varsity Tutors) is an AWS-first learning marketplace, so SQL Server is almost always the *legacy / Microsoft-heritage* engine rather than the default for greenfield services — carried over from acquisitions or older internal tooling, run on **RDS for SQL Server** (or EC2) so it coexists with the AWS stack. The places an org genuinely *keeps* SQL Server, rather than migrating, are where the value is in the Microsoft data ecosystem: **SSRS** (reporting), **SSIS** (ETL/integration), heavy **T-SQL business logic baked into stored procedures**, and analyst/finance reporting workflows. The realistic posture: meet the legacy system where it is, be fluent enough in T-SQL and execution plans to keep it healthy, and migrate *deliberately* — the stored-proc logic and SSRS reports are the hard, sticky part, not the table data.

### Graduated interview questions + model answers

**Q1 (basic) — "What's the difference between a clustered and a nonclustered index in SQL Server?"**
*Model answer / talking points:* "A clustered index physically orders the table's rows by its key — the data pages *are* the leaf level of that index, so the clustered index essentially *is* the table. You get exactly one per table. A nonclustered index is a separate B-tree holding its key columns plus a locator back to the row. The practical consequence: queries that filter or range-scan on the clustered key are very cheap, while a nonclustered index that doesn't cover the query has to do a key lookup back into the clustered index per row." Hook: "Coming from MySQL, the trap is that InnoDB clusters on the PK *for* you, but SQL Server lets you choose the clustered key independently of the primary key."

**Q2 (mechanism) — "What does INCLUDE do on a nonclustered index, and why would I use it?"**
*Model answer / talking points:* "INCLUDE adds columns to the *leaf pages* of a nonclustered index without making them part of the sortable key. The point is to make the index **covering** — so a query is satisfied entirely from the index and skips the **key lookup** back into the clustered index. If I have an index on `TutorId` but my query also selects `StartsAt` and `Status`, without INCLUDE the engine seeks the index and then does one extra read per row to fetch those columns; with `INCLUDE (StartsAt, Status)` they're already in the leaf, so the lookup disappears. I use INCLUDE for columns I only need to *return*, and key columns for what I need to *seek or sort on* — INCLUDE'd columns are payload, not seekable, and don't bloat the B-tree's intermediate levels the way key columns do."

**Q3 (intermediate) — "Compare SQL Server's default isolation behavior to MySQL's. Why might the same query block in one and not the other?"**
*Model answer / talking points:* "SQL Server defaults to READ COMMITTED implemented with **locking** — reads take shared locks, so a reader can block a writer and vice versa. InnoDB defaults to **REPEATABLE READ** backed by **MVCC**, where reads hit a consistent snapshot and don't take those shared locks, so the same workload blocks far less. To get MySQL-like non-blocking reads in SQL Server you turn on **READ_COMMITTED_SNAPSHOT (RCSI)** at the database level — then READ COMMITTED reads a **row version** instead of taking a shared lock, so readers stop blocking writers. The tradeoff is the version store, which lives in **tempdb**: you've moved the cost from lock contention to tempdb I/O and growth. So 'why does it block here but not there' is usually 'SQL Server is doing pessimistic locking and the other engine is doing MVCC.'"

**Q4 (advanced / GOTCHA) — "A stored procedure was fast for weeks, then suddenly got slow with no code change. What happened?"**
*Model answer / talking points:* "That's the signature of **parameter sniffing** with a bad cached plan. When the proc compiles, the optimizer sniffs the parameter values from *that* call and caches a plan tuned for them. After a restart, a stats update, or a memory-pressure eviction, the proc recompiles — and if the *first* call to repopulate the cache happens to pass an unrepresentative value, you cache a plan that's great for that value and terrible for the common case. The classic shape: a nested-loop/seek plan gets cached for a tutor with a handful of sessions, then reused for one with millions, where a hash join and scan would've been right. Fixes depend on how surgical I want to be: `OPTION (RECOMPILE)` to recompile per execution (fresh plan, more CPU), `OPTIMIZE FOR (@p = <representative value>)` or `OPTIMIZE FOR UNKNOWN` to plan for an average rather than the sniffed value, or restructuring the proc. SQL Server 2022's Parameter Sensitive Plan optimization can cache multiple plans, but I'd still diagnose it by comparing the cached plan's **estimated vs actual rows** — a huge gap is the tell." *This is the single highest-value answer to nail — it's the SQL Server equivalent of GraphQL's N+1, and recognizing "fast then suddenly slow, no code change → parameter sniffing" instantly marks you as someone who's actually operated the engine.*

**Q5 (tradeoffs) — "When would you use a table variable versus a temp table, and what's the catch?"**
*Model answer / talking points:* "Both are scratch storage, but the optimizer treats them very differently. A **temp table (`#t`)** gets **statistics**, so the optimizer estimates its row count realistically and can build a good plan — and it can be indexed. A **table variable (`@t`)** historically gives the optimizer a **fixed estimate of one row** regardless of how many rows it actually holds, so on large sets you get plans built for one row applied to a million — catastrophic nested loops. So: small, known-tiny sets, or where I want to avoid recompiles and statistics overhead → table variable; anything large or where the optimizer needs to reason about cardinality → temp table. The nuance is that SQL Server 2019+ added **table-variable deferred compilation**, which lets it estimate real cardinality, and `OPTION (RECOMPILE)` has always sidestepped the 1-row problem — but I wouldn't rely on a table variable for a big intermediate result." Alternatives worth naming: **table-valued parameters** for passing sets into procs, CTEs when I don't actually need materialization.

**Q6 (judgment) — "We're AWS-first and lean toward MySQL/Aurora and DynamoDB. When is it right to keep something on SQL Server rather than migrate it?"**
*Model answer / talking points:* "I'd keep it on SQL Server when the value lives in the Microsoft data *ecosystem*, not just the table data — heavy **T-SQL stored-procedure business logic**, **SSRS** reports, **SSIS** ETL pipelines, or analyst/finance workflows built around those tools. Migrating the rows is easy; rewriting hundreds of stored procedures and re-platforming reporting is the expensive, risky part, and it's pure cost with no new product value. So the call is economic: if the system is stable, the logic is sticky, and there's no scaling or licensing pain forcing the issue, I leave it on **RDS for SQL Server** where it coexists with the AWS stack and migrate deliberately — strangler-pattern the *new* surfaces onto Aurora/Dynamo while the legacy core keeps running. I'd prioritize migration when SQL Server licensing cost, operational burden, or a feature ceiling actually justifies the rewrite — not just because it's the odd engine out. The mistake is a big-bang migration that re-implements years of battle-tested T-SQL for no user-facing gain." *Ending on "migrate deliberately, the stored-proc/SSRS logic is the sticky part" is exactly the judgment the question is testing.*

### One-liners to have ready
- "The clustered index *is* the table — and unlike InnoDB, SQL Server lets you choose its key independently of the primary key."
- "INCLUDE makes an index covering — it kills the key lookup; key columns you seek on, INCLUDE'd columns you just carry."
- "SQL Server's default READ COMMITTED *locks*; turn on RCSI and readers stop blocking writers — paid for in tempdb."
- "SNAPSHOT is transaction-level (and can hit error 3960); RCSI is statement-level and conflict-free — don't conflate them."
- "Fast for weeks then suddenly slow with no code change? Parameter sniffing — a bad plan got cached."
- "Temp table gets statistics; table variable historically estimates one row — use `#temp` for big sets."
- "We keep SQL Server for the *ecosystem* — SSRS, SSIS, sticky T-SQL — and migrate deliberately, not big-bang."

## OAuth

> Prep note: You have real hands-on OAuth/OIDC experience, so this section is NOT here to re-teach you "delegated access" — skip the brochure. Its job is the senior-level precision an experienced dev is usually fuzzy on: the four roles and which grant maps to which client, why Implicit and Resource-Owner-Password are *deprecated* (not just "discouraged"), exactly how a resource server *validates* a JWT it didn't issue, the difference between scopes and claims, and the specific places PKCE/`state`/`redirect_uri` exact-match each defend. Interviewers separate "I've wired up Auth0/Cognito" from "I understand the protocol" by probing failure modes — token storage, open redirects, missing `state`. Name the mechanism and the threat it closes, not just the term.

### What you actually need to know (primer)

- **Four roles — get them exactly right, because every flow is just these four talking.** The **resource owner** is the human who owns the data (the student/parent). The **client** is the app asking for access (the web SPA, the iOS app, a backend service). The **authorization server (AS)** authenticates the owner, gets consent, and issues tokens (Cognito, Okta, Auth0, `accounts.google.com`). The **resource server (RS)** is the API holding the protected data and validating tokens (your scheduling/billing API). The classic blind spot: AS and RS are *different responsibilities* even in the same deployment — the AS mints tokens, the RS only *verifies* them. Conflating "logging in" (AS) with "accepting a bearer token" (RS) is the tell of someone who's only used a hosted SDK.

- **Grant types, and the one-line rule for which to use.** **Authorization Code + PKCE** is the modern default for *everything interactive* — web apps, SPAs, and native/mobile — full stop. **Client Credentials** is for machine-to-machine (a backend calling another with no user present; the app authenticates as *itself*). The **Refresh Token** grant exchanges a long-lived refresh token for a new access token without re-prompting. **Device Code** is for input-constrained devices (smart TV, CLI, a tutoring kiosk) — the device shows a code, you authorize on your phone. **Deprecated:** **Implicit** (`response_type=token`) returned the access token in the URL *fragment* — it leaks via browser history, referrer, and logs, and has no client auth; **Resource Owner Password Credentials (ROPC)** has the app collect the user's actual password, which defeats the entire point of delegation and is incompatible with federation/MFA. Both are dropped in OAuth 2.1. If someone proposes Implicit for a SPA, the answer is "Auth Code + PKCE — Implicit is dead."

- **Three token types — do not blur them.** The **access token** is a short-lived (minutes) **bearer** credential the client sends to the RS (`Authorization: Bearer <token>`); whoever holds it can use it, so it must be short-lived and transport-protected. The **refresh token** is long-lived, *never* sent to the RS — only to the AS's token endpoint — to mint fresh access tokens; it can (and for public clients should) be **rotated** on each use so a stolen one is detectable. The **OIDC ID token** is a **JWT about the user** (`sub`, `email`, `name`, `auth_time`) meant for the *client* to establish identity — it is **not** an API access credential. Sending an ID token to a resource server as if it were an access token is a common and wrong pattern.

- **JWT anatomy and how an RS actually validates one.** A JWT is `header.payload.signature`, three base64url segments. The header carries `alg` (e.g. `RS256`) and `kid` (key id). To validate a token it *didn't* issue, the RS: (1) fetches the AS's public keys from its **JWKS** endpoint — whose URL it gets from the AS's OIDC discovery document (`/.well-known/openid-configuration` → `jwks_uri`) — selecting the key by `kid`; (2) **verifies the signature**; (3) checks the **claims** — `iss` (issuer matches the AS you trust), `aud` (audience is *this* API), `exp`/`nbf` (not expired / not-before), and often `scope`. Two senior gotchas: **never trust `alg` from the token blindly** — pin the expected algorithm, because the classic attack is forging `alg:none` or downgrading RS256→HS256 to abuse the public key as an HMAC secret; and a JWT is **stateless**, so it stays valid until `exp` even after logout — revocation needs short lifetimes plus introspection/denylist if you need an instant kill.

- **Opaque tokens vs JWTs, and when you DON'T self-validate.** Access tokens come in two flavors. A **JWT (self-contained)** the RS validates locally via signature — fast, no network call. An **opaque/reference token** carries no info; the RS calls the AS's **introspection endpoint** (RFC 7662) to ask "is this active, and what's it good for?" — slower but instantly revocable. Rule of thumb: you validate tokens *you* are the audience for, via signature (JWT) or introspection (opaque). You generally **don't** crack open or trust a third party's access token — to a relying party, another provider's access token is opaque by design.

- **Scopes are NOT claims.** **Scopes** are coarse, *requested* permissions that bound what the token may do (`read:sessions`, `billing:write`) — authorization. **Claims** are *assertions* inside the token (`sub`, `email`, `roles`, `tenant_id`) — attributes/identity. Scopes set the ceiling; claims describe the subject. A frequent muddle: doing fine-grained per-record authorization purely with scopes — scopes cap the reach, but row-level decisions ("can THIS parent see THIS student") belong in the RS's logic using claims, not in an ever-growing scope list.

- **The Authorization Code + PKCE flow, step by step, and what each guard defends.** (1) The client generates a random `code_verifier`, derives `code_challenge = BASE64URL(SHA256(code_verifier))` (the S256 method), and redirects the browser to the AS `/authorize` with `response_type=code`, `client_id`, `redirect_uri`, `scope`, a random **`state`**, and the `code_challenge`. (2) The AS authenticates the user, gets consent, and redirects back to the **exact** `redirect_uri` with a short-lived **authorization code** and the echoed `state`. (3) The client **verifies `state` matches** (the **CSRF** defense — drop the response if not), then POSTs the code + raw `code_verifier` to `/token`. (4) The AS recomputes `BASE64URL(SHA256(code_verifier))`, compares it to the stored `code_challenge`, and only then returns the tokens. **PKCE** binds the code to the client that started the flow, so an intercepted code (e.g. a malicious app grabbing a mobile redirect) is useless without the verifier. **`redirect_uri` exact-match** (not prefix/wildcard) is what stops an attacker steering the code to a host they control.

- **Where this actually shows up at a company like Nerdy.** A learning marketplace has students, parents, tutors, and staff across a web SPA and native mobile apps on an AWS-first stack — so the realistic picture is **Cognito** (or Okta/Auth0) as the AS, **Authorization Code + PKCE** for *all* of those interactive clients (no Implicit anywhere), and many backend services as resource servers validating JWTs locally against Cognito's JWKS — checking `iss`, `aud`, and a `scope`/`groups` claim. **Federation** matters: "Sign in with Google/Microsoft," plus school/district **SSO via SAML or OIDC**, so a parent or partner district logs in with their existing identity. **Client Credentials** covers service-to-service and partner integrations. The cross-cutting design calls: token lifetime (short access tokens + rotating refresh tokens), where the SPA stores tokens, and keeping staff/admin scopes least-privilege.

### Graduated interview questions + model answers

**Q1 (basic) — "Walk me through OAuth's four roles and how OAuth differs from OIDC."**
*Model answer / talking points:* "Four roles: the **resource owner** owns the data; the **client** is the app requesting access; the **authorization server** authenticates the user and issues tokens; the **resource server** is the API that validates those tokens and serves the data. OAuth 2.0 is an **authorization** framework — it grants a client scoped, token-based access without handing over the password. **OIDC** is a thin **identity/authentication** layer on top that adds an **ID token** — a JWT telling the client *who* the user is. So OAuth answers 'can this app do X,' OIDC adds 'and here's who logged in.' The shallow tell is conflating the two, or assuming the AS and RS must be the same box — they're distinct responsibilities."

**Q2 (mechanism) — "How does a resource server validate a JWT access token it receives?"**
*Model answer / talking points:* "It splits `header.payload.signature`, reads the `kid` from the header, and fetches the matching public key from the AS's **JWKS endpoint** — whose URL comes from OIDC discovery — caching it and refreshing on rotation. It **verifies the signature**, then validates claims: `iss` is an issuer I trust, `aud` is *this* API, `exp`/`nbf` are in range, and the required `scope` is present. Crucially I **pin the expected algorithm** rather than trusting the token's `alg` — otherwise you're open to `alg:none` or an RS256→HS256 downgrade where the public key gets abused as an HMAC secret. And JWTs are stateless: valid until `exp`, so logout/revocation needs short lifetimes or an introspection/denylist. I only self-validate tokens where I'm the audience — a third party's access token is opaque to me."

**Q3 (intermediate) — "A new SPA needs login. Which grant, and why not the others?"**
*Model answer / talking points:* "**Authorization Code with PKCE.** PKCE was originally for mobile but is now standard for SPAs and even confidential web clients — it binds the authorization code to the client instance via the `code_verifier`/`code_challenge` pair, so an intercepted code can't be redeemed. I would **not** use **Implicit** — it's deprecated; it returned the token in the URL fragment where it leaks through history, referrer, and logs, and OAuth 2.1 drops it. I would **not** use **ROPC** — having the SPA collect the user's password kills delegation and breaks federation and MFA. Client Credentials doesn't apply because there's a real user present, not a machine. So: Auth Code + PKCE, short-lived access token, rotating refresh token."

**Q4 (advanced / GOTCHA) — "Walk me through the full Authorization Code + PKCE flow, and tell me what `state`, PKCE, and `redirect_uri` matching each protect against."** *(highest-value answer to nail — this is the OAuth equivalent of GraphQL's N+1; nailing it proves you understand the protocol, not just an SDK)*
*Model answer / talking points:* "The client makes a `code_verifier`, computes `code_challenge = BASE64URL(SHA256(verifier))`, and redirects to `/authorize` with `response_type=code`, `client_id`, an exact `redirect_uri`, `scope`, a random **`state`**, and the `code_challenge`. The AS authenticates the user, gets consent, and redirects back to the **exact** registered `redirect_uri` with a short-lived **code** and the echoed `state`. The client **checks `state` matches** what it sent, then exchanges the code plus the raw **`code_verifier`** at `/token`; the AS hashes and base64url-encodes the verifier, compares it to the stored challenge, and only then issues tokens. Three guards, three threats: **`state`** is the **CSRF** defense at the authorization endpoint — reject the callback if it doesn't match; **PKCE** defeats **authorization-code interception** (a malicious app grabbing a mobile redirect, or a leaked code) because the code is worthless without the verifier; **`redirect_uri` exact-match** prevents the **open-redirect / code-exfiltration** attack where an attacker steers the code to a host they control. They're layered, not redundant — skip `state` and you have CSRF; allow loose `redirect_uri` matching and you have token theft."

**Q5 (tradeoffs) — "Where does a SPA store its tokens, and what's the tradeoff?"**
*Model answer / talking points:* "It's pick-your-poison between two threats. **`localStorage`** is easy and survives reloads but is fully readable by JavaScript, so any **XSS** drains the token instantly. An **httpOnly, Secure, SameSite cookie** is invisible to JS so XSS can't read it, but a cookie is sent automatically, which reintroduces **CSRF** — so you pair it with SameSite and/or anti-CSRF tokens. The increasingly standard pattern is the **BFF (backend-for-frontend)**: the SPA holds *no* tokens; a thin server keeps them in a server-side session and proxies API calls, so the browser only ever has a hardened session cookie. The general rule: keep access tokens short-lived, refresh tokens out of JS reach, and never put a token anywhere a URL or log can capture it. Naming both the XSS and CSRF sides — and the BFF as the escape hatch — is what shows depth."

**Q6 (judgment) — "Tutors authenticate fine, but a security review flags 'token leakage.' Where do you look, and how do you reason about revocation?"**
*Model answer / talking points:* "First, the **leak surfaces**: bearer tokens showing up in **server/access logs**, in **URLs/query strings** (so they leak via Referer and proxies — tokens belong in the `Authorization` header or body, never the URL), in **browser history** (the Implicit-flow problem), or echoed in error pages. I'd grep logs/APM for `Bearer`, scrub them, and confirm we're not on a deprecated flow. Then **revocation reasoning**: plain JWTs are stateless and live until `exp`, so a leaked token is valid until then — mitigations are **short access-token lifetimes**, **refresh-token rotation** (a stolen refresh token gets detected on reuse and the whole token family is revoked), the **revocation endpoint** (RFC 7009) for refresh tokens, and **introspection** or a short-TTL denylist if I need *instant* invalidation. The honest tradeoff is stateless-and-fast (JWT, no lookup) versus instantly-revocable (opaque + introspection); for high-sensitivity actions like billing or staff admin I'd lean toward shorter lifetimes or introspection."

## REST

> Prep note: You have real hands-on REST experience, so this section skips the "what is a resource" basics and goes straight for the senior depth where experienced devs get fuzzy: the safe/idempotent matrix and why retries hinge on it, idempotency *keys* for POST, the status codes that separate seniors (201 vs 202, 400 vs 422, 401 vs 403, 404 vs 409, 429 + `Retry-After`), the honest truth that almost every "REST" API is really HTTP-RPC (Richardson level 2, no HATEOAS), conditional requests / optimistic concurrency with ETags, and the offset-vs-cursor pagination trap. Your edge in the room isn't reciting verbs — it's naming the *mechanism* behind a correct choice ("relative PATCH isn't idempotent, so the gateway must not blind-retry it"). Confident, precise, mechanism-first.

### What you actually need to know (primer)

- **The safe/idempotent matrix is the load-bearing mental model, and retries depend on it.** *Safe* = no observable server state change (read-only). *Idempotent* = doing it N times leaves the same server state as doing it once (the *response* can differ; the *effect* can't). The matrix: **GET/HEAD/OPTIONS = safe + idempotent. PUT/DELETE = idempotent but NOT safe. POST = neither. PATCH = NOT safe and (per RFC 5789) NOT guaranteed idempotent** — it *can* be if the patch is absolute (`set status=active`), but isn't if it's relative (`increment balance by 10`). This matters because every retry layer — browser, load balancer, SDK, an SQS/Lambda retry, a service mesh like Envoy — can *automatically* retry idempotent methods on a timeout, and *must not* auto-retry POST or relative PATCH: a timeout means "I don't know if it landed," and re-sending a non-idempotent request risks double-charging. DELETE has a subtlety: it's idempotent on *state* (the row is gone either way), but the second call typically returns 404, not 204 — same effect, different response, still idempotent.

- **Idempotency keys are how you make POST safe to retry — this is the payments answer.** POST isn't idempotent by spec, but you can *engineer* idempotency: the client generates a unique key (UUID) per logical operation and sends it as `Idempotency-Key: <uuid>`. The server stores `key -> (status, response)` in a durable table with a uniqueness constraint; on a replay of the same key it returns the *original* recorded response instead of re-executing. This is how Stripe's API works. The gotchas seniors miss: the key must cover the *intent*, so the same key with a different body should error, not silently succeed (Stripe does exactly this); you need a TTL (Stripe's is 24h) and a race-safe "in-progress" state so two concurrent retries don't both execute; and the key must be persisted *before* the side effect, not after.

- **Status codes are a programmatic API surface, and the distinctions are where seniority shows.** The ones that separate a senior: **201 Created** (resource made — include a `Location` header) vs **200 OK** vs **202 Accepted** (accepted for *async* processing, not done yet — return a status URL) vs **204 No Content** (success, empty body — typical for PUT/DELETE). **400 Bad Request** (malformed/unparseable — JSON won't parse, wrong type) vs **422 Unprocessable Entity** (syntactically fine but semantically invalid — valid JSON, but `age = -5` or a failed business rule). **401 Unauthorized** (not authenticated — *who are you?*, triggers re-login) vs **403 Forbidden** (authenticated but not allowed — *I know who you are, and no*). **404 Not Found** vs **409 Conflict** (request collides with current state — duplicate create, edit conflict, version mismatch). **429 Too Many Requests** (rate limited) and **503 Service Unavailable**, both of which should carry **`Retry-After`**. Mapping the wrong code is a real bug: returning 200 with an error body, or 500 for a client validation failure, breaks every retry and alerting heuristic built on top of you.

- **Richardson Maturity Model — and the honest fact that your "REST" API is level 2.** Level 0 = one URL, one verb (HTTP as a dumb tunnel, e.g. SOAP-style POST-everything). Level 1 = resources (many URLs). Level 2 = HTTP verbs + status codes used correctly (GET reads, POST creates, 201/404/etc.). Level 3 = **HATEOAS** (Hypermedia As The Engine Of Application State): responses embed links telling the client what it can do next, so it navigates by following links instead of hardcoding URL templates. The senior-honest truth: **the vast majority of production "REST" APIs are level 2 — HTTP-as-RPC with nice URLs — and full HATEOAS is rare**, because typed clients/SDKs and OpenAPI specs deliver most of the discoverability value with far less ceremony. Knowing this lets you say "we're pragmatically level 2, and that's a deliberate, common choice" instead of pretending to Fielding-pure REST.

- **Conditional requests give you free caching *and* optimistic concurrency — same `ETag` mechanism, two payoffs.** The server returns an `ETag` (an opaque version hash) and/or `Last-Modified` on a resource. **Caching:** the client later sends `If-None-Match: "<etag>"`; if unchanged the server returns **304 Not Modified** with no body, saving bandwidth. **Optimistic concurrency (the lost-update fix):** on a write the client sends `If-Match: "<etag>"`; if the resource changed since they read it, the server returns **412 Precondition Failed** instead of clobbering someone else's edit. `Cache-Control` (`max-age`, `no-cache`, `private`/`public`, `stale-while-revalidate`) governs *whether/how long* to cache; ETag governs *revalidation*. This is the HTTP-native caching story REST gets "for free" and GraphQL gives up — and it's exactly why you keep public read-heavy content on REST URLs.

- **Pagination: offset/limit is the easy default and the wrong one at scale — cursor/keyset is the senior answer.** `?limit=20&offset=10000` makes the database walk and discard 10,000 rows before returning 20 (`OFFSET` cost grows with the offset, so it gets *slower the deeper you page*), and it's **unstable under concurrent inserts/deletes**: if a row lands near the front while a user pages, page 2 repeats or skips a row. **Cursor (keyset) pagination** instead encodes "where you left off" — `WHERE (created_at, id) < (:last_ts, :last_id) ORDER BY created_at, id LIMIT 20` — which rides the index, so per-page cost stays flat no matter how deep you go, and it's stable under inserts. The cost: you lose random-access "jump to page 500" and total counts get awkward. Rule of thumb: offset is fine for small, bounded, admin-style lists; cursor is the answer for large, hot, or infinitely-scrolling feeds (most mobile UX).

- **Versioning, error bodies, and rate limiting are the maintainability trio.** *Versioning:* URI versioning (`/v1/users`) is ugly-but-honest and the most common because it's trivial to route, cache, and curl; header versioning (`Accept: application/vnd.nerdy.v2+json`) is "cleaner" but harder to test and cache; the real senior move is *additive, non-breaking evolution* (add fields, never repurpose or remove) so you rarely cut a v2 at all. *Errors:* return a **consistent machine-readable shape** — **RFC 9457 `application/problem+json`** (`type`, `title`, `status`, `detail`, `instance`), which obsoletes RFC 7807 — so clients branch on a stable field instead of regex-ing prose. *Rate limiting:* `429` plus `Retry-After` and `X-RateLimit-Limit/Remaining/Reset` (or the newer standardized `RateLimit` headers) so well-behaved clients back off instead of hammering. *Content negotiation:* `Accept`/`Content-Type` let one resource serve JSON, CSV, or a different schema version.

- **Where this actually shows up at a company like Nerdy.** Nerdy (Varsity Tutors) is a learning marketplace — students, tutors, parents, staff, web + native mobile, AWS-first, many backend services — and REST is the default at every public edge: account/profile CRUD, scheduling and booking, the course/content catalog, billing. The senior decisions are concrete: **payment endpoints need idempotency keys** because mobile networks drop and the app retries (double-charging a parent is a Sev-1); **session/feed lists need cursor pagination** because offset dies on hot, frequently-written tables and breaks infinite scroll; **concurrent edits to a tutor's availability or a shared booking need `If-Match`/412** to avoid lost updates; **public read-heavy catalog content rides `Cache-Control` + ETag through CloudFront** so it caches at the CDN; and an **API Gateway / ALB in front must only auto-retry idempotent methods** — a blind retry of POST through the gateway is how you ship a duplicate-booking bug. Internally those services may speak gRPC to each other, but the browser/mobile contract is REST.

### Graduated interview questions + model answers

**Q1 (basic) — "What does it mean for an HTTP method to be safe versus idempotent, and which methods are which?"**
*Model answer / talking points:* "Safe means no observable state change — purely a read. Idempotent means calling it N times leaves the server in the same state as calling it once; the response can differ but the effect can't. GET, HEAD, and OPTIONS are safe and idempotent. PUT and DELETE are idempotent but not safe — they change state, but repeating them lands you in the same place. POST is neither. And PATCH is not safe and *not* guaranteed idempotent — it depends on whether the patch is absolute or relative." Hook: "DELETE is the classic gotcha — deleting twice is idempotent on *state* even though the second call usually returns 404 instead of 204."

**Q2 (mechanism) — "Why do those properties actually matter? Walk me through what they control."**
*Model answer / talking points:* "They control what's safe to *retry*. Every layer in the stack — browser, ALB, SDK, an SQS or Lambda retry, a service mesh — will automatically retry idempotent methods on a timeout, because a timeout means 'I don't know if it landed.' For GET or PUT that's free; for POST it's dangerous, because re-sending might double-apply the effect — a second charge, a second booking. So the safe/idempotent matrix is really the contract that tells every retry mechanism what it's allowed to do automatically. The corollary: your gateway config must *not* blind-retry POST or relative PATCH."

**Q3 (intermediate) — "Tell me about status codes that people get wrong. Distinguish a few pairs."**
*Model answer / talking points:* "The pairs I watch for: **201 vs 200 vs 202** — 201 for a created resource with a `Location` header, 202 when I've accepted work for *async* processing that isn't done yet, 200 for a plain successful read or update. **400 vs 422** — 400 is malformed, the body won't even parse; 422 is well-formed but semantically invalid, like a negative age or a failed business rule. **401 vs 403** — 401 is 'I don't know who you are,' which should trigger re-auth; 403 is 'I know exactly who you are, and you're not allowed.' **404 vs 409** — 409 is a conflict with current state, like a duplicate create or an edit collision. And **429** for rate limiting, ideally with `Retry-After`. This matters because status codes are a *programmatic* surface — clients, retries, and alerting all branch on them — so returning 200 with an error body, or a 500 for a validation failure, is a real bug, not a cosmetic one."

**Q4 (advanced / GOTCHA — the highest-value answer to nail) — "A mobile client POSTs a payment, the network times out, and the app retries. How do you avoid double-charging?"**
*Model answer / talking points:* "This is the POST-isn't-idempotent problem, and the fix is an **idempotency key**. The client generates a UUID per logical payment and sends it as an `Idempotency-Key` header. The server has a durable table keyed on that value with a uniqueness constraint; before charging, it records the key in an 'in-progress' state, and on any replay of the same key it returns the *original* recorded response instead of charging again. That turns a non-idempotent POST into something safe to retry through a flaky network — it's exactly how Stripe does it. The subtleties I'd call out: persist the key *before* the side effect, not after; make same-key-different-body an error so you don't silently mask a bug; handle the concurrent-retry race so two in-flight retries don't both execute; and put a TTL on the key. Without this, every gateway and SDK retry is a latent double-charge." *(Flag: this is the single highest-value answer — it's REST's equivalent of GraphQL's N+1. Naming 'idempotency key' plus the persist-before-side-effect detail proves you've shipped this, not just read about it.)*

**Q5 (tradeoffs) — "How would you paginate a large, frequently-updated list — say a tutor's session feed — and what's wrong with the obvious approach?"**
*Model answer / talking points:* "The obvious approach is `limit`/`offset`, and it has two real problems at scale. First, performance: `OFFSET 10000` makes the database walk and throw away 10,000 rows before returning your page, so it gets *slower the deeper you page* — cost grows with the offset. Second, correctness: it's unstable under concurrent inserts and deletes — if a row is added near the front while someone's paging, they'll see a duplicate or skip a row across page boundaries. The senior answer is **cursor / keyset pagination**: encode where you left off as `WHERE (created_at, id) < (:last_ts, :last_id) ORDER BY created_at, id LIMIT n`. That rides the index, so per-page cost stays flat no matter how deep, and it's stable under inserts — exactly what an infinite-scroll mobile feed wants. The tradeoff is you give up random-access 'jump to page 500' and clean total counts. So: offset for small bounded admin lists, cursor for hot or infinite feeds."

**Q6 (judgment) — "Most 'REST' APIs aren't really RESTful by Fielding's definition. Does that matter, and how do you decide where to put rigor?"**
*Model answer / talking points:* "Right — by the **Richardson Maturity Model**, most production 'REST' APIs are level 2: nice resource URLs plus correct verbs and status codes, but no HATEOAS, which is what level 3 requires. And honestly that's fine — full hypermedia is rare because typed SDKs and OpenAPI specs deliver most of the discoverability benefit with far less ceremony, so calling it 'HTTP-RPC with good hygiene' is accurate, not an insult. Where I *do* spend rigor is the stuff that bites in production: correct verb semantics so retries are safe, accurate status codes because clients branch on them, idempotency keys on money-moving POSTs, `If-Match`/412 for concurrent edits, ETags and `Cache-Control` so read-heavy content caches at the CDN, cursor pagination on hot lists, and a consistent error shape like RFC 9457 problem+json. That's the pragmatic line: skip HATEOAS purity, never skip the mechanisms that determine whether retries, caching, and concurrency behave correctly. And sometimes the honest move is to admit an endpoint is really an RPC call — `POST /sessions/{id}:start` — and that's okay; forcing every action into a resource verb is its own anti-pattern."

- "POST isn't idempotent; an idempotency key is how you *make* it safe to retry — that's the payments answer."
- "Status codes are an API, not decoration — 401 is 'who are you?', 403 is 'I know, and no'; 400 is malformed, 422 is invalid."
- "Most 'REST' is Richardson level 2 — HTTP-RPC with nice URLs — and shipping level 2 on purpose is fine."

## Webhooks

> Prep note: You have real hands-on webhook experience, so this section skips the "outbound callback vs polling" basics — the interviewer assumes you know that. Your job is senior depth on the parts experienced devs are routinely fuzzy on: verifying HMAC signatures *correctly* (sign over the **raw** body, constant-time compare, secret rotation), replay defense via a signed timestamp + tolerance window, idempotency under **at-least-once, out-of-order** delivery, the "2xx-fast-then-queue" discipline and *why* the provider punishes you for violating it, and the symmetric problem on the **producing** side (transactional outbox, retry with backoff + jitter, dead-lettering). The highest-leverage move is to frame webhooks as a *distributed-systems reliability problem*, not an HTTP endpoint — name the failure modes (retry storms, duplicate processing, thundering herd) and the mechanism that defends each. Confident register: "delivery is at-least-once, so I make the consumer idempotent and the producer durable."

### What you actually need to know (primer)

- **Verify the signature over the RAW request body, before any parsing.** The provider HMACs the exact bytes it sent — `HMAC-SHA256(secret, raw_bytes)` — and puts the digest in a header (`Stripe-Signature`, `X-Hub-Signature-256`, etc.). You must recompute over the *identical* bytes. The classic bug: your framework parses JSON, then you re-serialize to verify — but `JSON.parse` → `JSON.stringify` reorders keys, drops whitespace, and rewrites number/unicode formatting, so your bytes differ from theirs and verification fails (or worse, you "fix" it by trusting unverified input). In Express, capture the raw buffer (`express.raw()` or a `verify` callback) *before* `express.json()` touches it. Mental model: **the signature authenticates bytes on the wire, not your in-memory object.**

- **Compare signatures in constant time.** A naive `===` on the hex digest short-circuits on the first mismatching byte, leaking timing information that can let an attacker recover a valid signature byte-by-byte. Use a timing-safe compare: Node's `crypto.timingSafeEqual(Buffer.from(a), Buffer.from(b))` — it throws if the buffers differ in length, so length-check first. A small detail that signals you understand the attack surface, not just "we check the signature."

- **The shared-secret model — and the asymmetric variant.** Most webhooks use a **symmetric** HMAC: the same signing secret lives on both the provider and your server, so *anyone who can verify can also forge*. That's acceptable because the secret never leaves your two systems — but it means you treat it like a credential (Secrets Manager / vault, never in source). Some providers offer **asymmetric** signing (e.g. Svix supports Ed25519, though its default is still HMAC): they sign with a private key, you verify with their public key, so a leaked *verification* key can't forge. Know both exist; the default you'll meet — Stripe, GitHub's `X-Hub-Signature-256`, Slack — is HMAC with a shared secret.

- **Secret rotation needs overlapping keys.** You can't atomically swap a shared secret across two systems without dropping in-flight events. The pattern: the provider keeps *multiple active signing secrets* during a rotation window, signing each event with **all** of them; your verifier accepts a signature matching **any** currently-valid secret (try old and new), then you retire the old one once cutover is confirmed. Stripe is concrete here — rolling a secret keeps the old one valid for up to 24 hours and signs every event with both, so you deploy the new secret, watch for verification failures, then drop the old. If your verifier only ever checks one secret, rotation becomes an outage.

- **Replay defense = a signed timestamp + a tolerance window.** A captured-and-replayed payload still carries a valid signature, so HMAC alone doesn't stop replay. Providers fold a **timestamp into the signed material** — Stripe signs `{timestamp}.{raw_body}` (period-joined) and ships the timestamp as `t=` in `Stripe-Signature` — and you reject anything where `|now − t|` exceeds a tolerance (Stripe's libraries default to **300 s / 5 min**). The window is wide enough to absorb clock skew and legitimate retries, narrow enough to limit a stolen payload's lifetime. Belt-and-suspenders: pair it with event-id dedup so even an in-window replay is processed once.

- **Delivery is AT-LEAST-ONCE → duplicates AND reordering: dedup by event id, never trust order.** The provider retries on any non-2xx or timeout, so the *same* event can arrive 2+ times; and because retries interleave with fresh deliveries, event #5 can land before event #4. Two consequences: (1) make the handler idempotent by recording the provider's unique **event id** in a dedup store (a DB unique constraint, or a Redis key with TTL) and short-circuiting on a repeat; (2) never reconstruct state from arrival order — reconcile against the resource's current state, a version/sequence number in the payload, or a re-fetch from the provider's API to get truth. "I assumed webhooks arrive in order" is a real, painful production bug.

- **Return 2xx within seconds, then process async — and know *why*.** The provider reads a slow response or any non-2xx as a failed delivery and **retries**. If your handler does real work synchronously (DB writes, downstream calls, email) and is slow, you get timeouts → retries → the *same* slow work runs again → duplicate side effects and a retry storm that takes you down exactly when you're already struggling. The discipline is **store-then-ack**: persist the raw, verified event durably, return 2xx immediately, do the work on a queue. The HTTP handler's only job is "I have safely received this."

- **Consumer reliability is store-then-process with a dead-letter.** Persist the raw event (so you can replay/debug, and so the ack is honest), enqueue it, and let a worker process it. At Nerdy that queue is **SQS / SNS / EventBridge**; failed processing retries with backoff, and after N attempts the message lands in a **dead-letter queue (DLQ)** for inspection instead of looping forever or being silently dropped. The DLQ is what turns a poison event from an outage into an alert.

- **Producer side = transactional outbox + retry with backoff *and jitter* + a delivery log.** If *you* emit webhooks, the hard part is the dual-write: "I committed the state change but crashed before sending" (or vice versa). The **transactional outbox** pattern fixes it — in the *same* DB transaction that changes state, you insert an `outbox` row; a separate relay reads the outbox and delivers, so the event is sent iff the state change committed. Deliver with **exponential backoff + jitter** (jitter is essential — synchronized retries cause a **thundering herd** that hammers a recovering consumer in lockstep), keep a **delivery log / attempt history**, and expose manual re-send. This is the producer mirror of the consumer's idempotency.

- **Verify by signature, not by IP allowlist (though defense-in-depth can pin both).** IP allowlisting is brittle — providers rotate and expand egress ranges (often a published but changing CIDR list), deliver through CDNs/proxies, and an IP says nothing about payload integrity. The cryptographic signature is the real authentication: it proves both *origin* and *that the body wasn't tampered with*. Pinning published IP ranges **in addition to** signature checking is fine; **instead of** is not. "We only check the source IP" is a weak answer.

- **The bootstrapping handshake — is the endpoint even real?** Many providers won't start sending until they confirm you control the URL: a **challenge-response** on registration (Slack's `url_verification` and Twitch EventSub send a `challenge` you echo back; Meta sends `hub.challenge` on a GET subscription-verification request), or signed verification events. Know this exists so "the webhook isn't firing" doesn't stump you — sometimes the subscription was simply never verified.

- **Where this shows up at a company like Nerdy.** A learning marketplace lives on third-party webhooks: **Stripe** for billing/subscription lifecycle (`invoice.paid`, `customer.subscription.deleted`), disputes/refunds, **calendar/video** providers (Zoom/Twilio) for session-started / recording-ready events, **CRM and email/SMS** delivery status, and identity/SSO events. AWS-first, the canonical shape: API Gateway / a thin Lambda or service verifies the signature, drops the raw event onto **SNS→SQS or EventBridge**, returns 2xx fast, and downstream consumers (billing, notifications, scheduling) process idempotently with a DLQ. Internally, that *same* event-driven backbone (SNS/SQS/EventBridge) is how Nerdy's own services fan out domain events — so webhook discipline (idempotency, ordering-agnostic, DLQ) is the same muscle as the internal event bus.

### Graduated interview questions + model answers

**Q1 (basic) — "Webhooks vs polling — when do you reach for a webhook, and what do you give up?"**
*Model answer / talking points:* "A webhook is push: the provider calls me the moment an event happens, so I get low latency and I'm not burning requests polling for changes that mostly aren't there. Polling is pull: simpler and stateless on my side, no public endpoint to secure, and it degrades gracefully if I'm down — I just catch up next poll. What I give up with webhooks is real: I now own a public, internet-facing endpoint, signature verification, and the whole at-least-once reliability problem. So I use webhooks for event-driven, latency-sensitive integrations like Stripe billing, and keep polling or a reconciliation job as a backstop for events I might have missed." *(SSE/WebSockets are the push-to-a-browser cousins — see Q6.)*

**Q2 (mechanism) — "Walk me through verifying an incoming webhook's HMAC signature. What's the most common way people get it wrong?"**
*Model answer / talking points:* "The provider HMACs the raw request body with a shared secret and sends the digest in a header. I recompute `HMAC-SHA256(secret, rawBytes)` and compare — and three things have to be right. First, the most common bug: I verify over the **raw bytes**, *before* any JSON parsing. Parse and re-serialize and key order and whitespace change, so my bytes no longer match theirs and verification breaks; in Express I capture the raw buffer before the JSON middleware runs. Second, I compare in **constant time** with `crypto.timingSafeEqual`, not `===`, so I don't leak a timing oracle. Third, the secret is a credential — it lives in Secrets Manager, and my verifier accepts **multiple secrets during rotation** so I can roll it without an outage. The shared-secret HMAC is symmetric, so verify-ability equals forge-ability — which is exactly why the secret never leaves my systems."

**Q3 (intermediate) — "HMAC proves the payload came from the provider. Why isn't that enough, and how do you stop replay attacks?"**
*Model answer / talking points:* "A valid signed payload stays valid forever, so anyone who captures one — a logged request, a leaked proxy buffer — can replay it and it'll verify. The defense is a **timestamp inside the signed material**: Stripe signs `{timestamp}.{body}`, and I reject anything where the timestamp falls outside a tolerance window — Stripe's default is about five minutes. The window trades clock skew and legitimate retries against how long a stolen payload stays useful. I pair it with **event-id dedup** so even a replay inside the window is processed at most once. Signature answers 'is it authentic and untampered,' timestamp answers 'is it fresh,' dedup answers 'have I already acted on it.'"

**Q4 (advanced / GOTCHA) — "Your handler ran the same Stripe event twice and double-charged a customer. What happened, and how do you make sure it never happens again?"**
*Model answer / talking points:* "This is the canonical webhook failure mode, straight out of **at-least-once delivery**. Stripe retries on any non-2xx or timeout, so if my handler did the work and was then slow to ack — or crashed after the side effect but before returning 2xx — Stripe re-delivers the *same* event and I run the side effect again. The fix is **idempotency keyed on the provider's event id**: before doing any work I record the event id behind a unique constraint (DB row or Redis key); a duplicate hits the constraint and I short-circuit with a 200. The dedup write and the side effect want to be in the same transaction (or the side effect itself idempotent — e.g. an idempotency key on the downstream Stripe call) so there's no gap. And I restructure to **store-then-ack**: persist the raw event, return 2xx fast, process on a queue — so slowness never triggers a retry in the first place. The deeper lesson: **never assume ordering or exactly-once** from webhooks; design the consumer to be safely repeatable and order-independent." *(This is the single highest-value answer to nail — the webhook equivalent of GraphQL's N+1. "At-least-once → idempotent by event id → store-then-ack" is the whole game.)*

**Q5 (tradeoffs) — "You're on the producing side now — your service emits webhooks to customers. What does reliable delivery require, and what are the tradeoffs?"**
*Model answer / talking points:* "First I have to solve the **dual-write problem**: I must not tell a customer about a state change that didn't commit, or commit one I never delivered. The **transactional outbox** handles that — in the same DB transaction as the state change, I insert an outbox row, and a separate relay delivers from it, so the event is sent if and only if the change committed. Delivery uses **exponential backoff with jitter** — jitter matters because without it, thousands of failed deliveries retry in lockstep and **thundering-herd** a consumer that's just coming back. I keep a **delivery log** with attempt history and signed payloads, **dead-letter** after N attempts, and expose manual re-send plus a way for customers to fetch missed events. The tradeoff is that I'm now running a durable, monitored delivery subsystem — real operational weight, which is why outbox + a queue (SQS/EventBridge) beats firing-and-forgetting from request handlers. And I sign every payload with a timestamp and unique event id so *my* consumers can do exactly the verification and dedup we just discussed."

**Q6 (judgment) — "Webhooks, SSE, WebSockets, polling — how do you choose? Where would each fit at Nerdy?"**
*Model answer / talking points:* "I reason about **who initiates, which direction data flows, and how long-lived the connection is.** **Polling** is client-pull with no persistent connection — simplest, fine for low-frequency needs or as a reliability backstop. **Webhooks** are server-to-**server** push over short-lived HTTP calls — the right tool for backend integrations like Stripe billing or Zoom recording-ready, where both ends are servers and I want durability and retries. **SSE** is server-to-**browser** push over one long-lived HTTP connection, unidirectional — great for a live feed, notifications, or 'tutor is online,' and it rides plain HTTP so it's simple and proxy-friendly. **WebSockets** are full-duplex over a persistent connection — the choice when the *browser* also needs to push continuously, like a live classroom whiteboard or real-time chat. So at Nerdy: webhooks for third-party/backend events, SSE for one-way live UI updates, WebSockets for interactive sessions, polling as the humble fallback. The trap is using a webhook where I needed a browser channel, or a WebSocket where SSE would've been half the complexity." *(Naming the initiator/direction axis is what separates this from a memorized list.)*

## GraphQL

> Prep note: You have zero hands-on GraphQL experience, and that's fine. You will NOT be asked to write a production resolver on the spot. You WILL be expected to (a) explain what GraphQL is and why it exists, (b) reason about its tradeoffs vs REST, and (c) show judgment about when it's the wrong tool. This section gets you to that bar. Be honest in the interview ("I haven't shipped GraphQL, but here's my mental model") — confident, accurate reasoning beats fake war stories.

### What you actually need to know (primer)

- **It's a typed query language for APIs, defined by a schema.** The schema is the contract between client and server. You declare *object types* with *fields*, built on scalars (`Int`, `Float`, `String`, `Boolean`, `ID`) plus `enum`, `interface`, `union`, and `input` types. Written in SDL (Schema Definition Language). Example: `type User { id: ID!, name: String!, sessions: [Session!]! }` — the `!` means non-null. Because the schema is typed and machine-readable, GraphQL gives you *introspection* (the API can describe itself), which powers autocomplete, docs, and tools like the GraphiQL/Apollo explorer.

- **The client asks for exactly the fields it wants, and gets back that exact shape.** You send a query; the JSON response mirrors it. Ask for `{ user { name } }` and you get just the name — not the whole user object. This "client-specified response shape" is the single most important idea in GraphQL and the root of most of its advantages.

- **Three operation types: queries, mutations, subscriptions.** A **query** reads data (like an HTTP GET). A **mutation** writes data — create/update/delete (like POST/PUT/PATCH/DELETE) — and can return the updated object in the same round trip. A **subscription** is a long-lived stream of server-pushed updates, normally over a WebSocket, for real-time features (live chat, a classroom whiteboard, "tutor is online"). Convention: mutations run sequentially, query fields resolve in parallel.

- **Resolvers are the functions that actually fetch each field's data.** The schema says *what* exists; resolvers say *how* to get it. Every field can have its own resolver, and they execute as a tree that matches the query's shape — e.g. resolve `user`, then for that user resolve `sessions`, then for each session resolve `tutor`. A resolver might hit a database, call a REST microservice, or read a cache. This is the bridge between the typed schema and your real backend.

- **One endpoint, not many.** A GraphQL API is typically a single URL (`POST /graphql`); the operation in the request body decides what happens. Contrast with REST, where each resource/shape gets its own URL (`/users/1`, `/users/1/sessions`, ...). One endpoint + client-chosen fields is the core structural difference from REST.

- **It directly solves over-fetching and under-fetching.** *Over-fetching* = a REST endpoint returns more than the screen needs (you call `/user/1` and get 30 fields to show a name and avatar). *Under-fetching* = one endpoint isn't enough, so the client makes several round trips and stitches the data together (get user, then get their sessions, then each session's tutor). GraphQL fixes both: one request, only the requested fields, with related/nested data fetched in the same call. This is especially valuable for **mobile** clients on slow or metered networks.

- **The N+1 problem is the classic GraphQL gotcha — know it cold.** Naïve resolvers fire one query for a list, then one *more* query per item in that list: fetch 10 sessions (1 query), then fetch each session's tutor individually (10 queries) = 11 ("N+1"). Under load this hammers your database. The standard fix is a **DataLoader** (per-request batching + caching): instead of resolving each tutor immediately, it collects all the requested tutor IDs in one tick of the event loop, issues a single batched query (`WHERE id IN (...)`), and caches results so duplicate IDs aren't re-fetched. Alternatives: resolve via SQL joins, or precompute. Just naming "N+1" and "DataLoader batching" signals real understanding.

- **Where a company like Nerdy would actually use it.** Nerdy (Varsity Tutors) is a learning marketplace with web + native mobile apps backed by many services — student/tutor profiles, scheduling and tutor availability, live session data, course catalog, messaging, reviews, billing/subscriptions. A GraphQL layer (often a "gateway" or BFF — backend-for-frontend) lets one flexible API aggregate all of those so each screen fetches exactly what it needs in one round trip, without the client juggling a dozen REST endpoints or the backend shipping a new endpoint for every UI change. Mobile bandwidth savings and faster product iteration are the practical wins; **subscriptions** fit live-classroom / real-time tutoring features. Expect them to run GraphQL in front of, not instead of, internal microservices that may still speak REST/gRPC to each other.

### Graduated interview questions + model answers

**Q1 (basic) — "What is GraphQL, in your own words?"**
*Model answer / talking points:* "GraphQL is a typed query language and runtime for APIs. Instead of many fixed REST endpoints, you expose a single endpoint backed by a schema that describes every type and field available. The client sends a query asking for exactly the fields it wants, and the response comes back in that same shape as JSON." Add the one-line hook: "The defining idea is that the *client* decides what data it gets, not the server." If you want a bonus point: "It was created at Facebook around 2012 and open-sourced in 2015, originally to let mobile apps fetch exactly what a screen needed in one request."

**Q2 (basic→intermediate) — "What's the difference between a query, a mutation, and a subscription? And what's a resolver?"**
*Model answer / talking points:* "A **query** reads data — think GET. A **mutation** changes data — create, update, or delete — like POST/PUT/DELETE, and it can return the changed object so the client doesn't need a second request. A **subscription** is a real-time stream: the server pushes updates to the client over a persistent connection, usually a WebSocket — good for live features. A **resolver** is the server-side function behind a field; the schema declares what fields exist, and each resolver knows how to actually fetch that field's value — from a database, another service, or a cache. The resolvers execute as a tree mirroring the query." Beginner-safe framing: "Schema = the menu; resolvers = the kitchen that makes each dish."

**Q3 (intermediate) — "How does GraphQL differ from REST? When is that difference an advantage?"**
*Model answer / talking points:* "Structurally: REST has many endpoints, each returning a fixed payload; GraphQL has one endpoint and the client specifies the response shape. The big practical advantage is killing **over-fetching** and **under-fetching** — instead of downloading fields you don't need, or making three round trips to assemble one screen, you make a single request for exactly what you need, including nested related data. You also get a strongly typed, self-documenting schema with introspection, which makes tooling and frontend/backend collaboration smoother. Where it shines: complex, data-rich UIs and mobile clients, and aggregating multiple backend services behind one flexible API." Show balance: "REST still wins on simplicity and HTTP-native caching — I'd weigh both."

**Q4 (intermediate→advanced) — "What is the N+1 problem in GraphQL and how do you solve it?"**
*Model answer / talking points:* "It comes from per-field resolvers. Say you query a list of 10 tutoring sessions and each session's tutor. The list resolver runs one query, then the tutor resolver runs once per session — that's 1 + 10 = 11 database calls, and it scales with the list size, which crushes the DB under load. The standard fix is a **DataLoader**: it batches all the individual tutor-ID lookups that happen in one event-loop tick into a single query like `SELECT * FROM tutors WHERE id IN (...)`, and caches per request so repeated IDs aren't fetched twice. Other options are resolving with SQL joins or denormalizing/precomputing. The key insight is that GraphQL's flexible nesting makes N+1 easy to introduce, so you design resolvers with batching in mind." This is the single highest-value answer to nail — it proves you understand GraphQL's real-world failure mode, not just the happy path.

**Q5 (tradeoffs) — "What are the downsides or hidden costs of GraphQL compared to REST?"**
*Model answer / talking points:* Name several concretely:
  - **Caching is harder.** REST leans on HTTP caching (URLs, `Cache-Control`, CDNs) almost for free. GraphQL's single POST endpoint with arbitrary query bodies doesn't cache at the HTTP layer the same way, so you push caching into the client (Apollo/Relay normalized cache) or add persisted queries / server-side caching.
  - **Server complexity & performance footguns** — resolvers, schema design, and N+1 require deliberate engineering (DataLoader, etc.).
  - **Security/abuse surface:** a client can craft expensive deeply-nested or recursive queries, so you need **query depth limiting, complexity analysis, timeouts, and persisted/allow-listed queries**; rate-limiting is also trickier than per-endpoint REST.
  - **Weaker fit for some things:** file uploads, binary data, and simple status/health endpoints are more natural over plain REST. Error handling is also different — GraphQL often returns HTTP 200 with an `errors` array rather than HTTP status codes.
  - **Learning curve & tooling buy-in** for the whole team. "So it's a real tradeoff: more frontend flexibility in exchange for more backend responsibility."

**Q6 (judgment) — "When would you NOT use GraphQL?"**
*Model answer / talking points:* "I'd avoid it when its strengths don't apply and its costs do:
  - **Simple APIs** with a few resources and one known client — REST is less overhead and easier to cache.
  - **CRUD or internal service-to-service** APIs where shapes are stable and predictable — GraphQL adds a layer for little benefit; REST or gRPC is often better.
  - **Cacheability is the priority** — public, read-heavy, CDN-cached content maps cleanly to REST URLs.
  - **File uploads / binary / streaming downloads**, or webhooks and simple health checks — more natural over plain HTTP.
  - **Small team / tight timeline** without GraphQL experience — the schema, resolver, and caching machinery may not pay for itself yet.
The honest summary: GraphQL earns its keep when you have many clients (especially mobile), complex and evolving data needs, or you're aggregating multiple services. If you don't have those pressures, REST is usually simpler and cheaper." Ending on "right tool for the problem" judgment is exactly what the *when-NOT* question is testing.

### One-liners to have ready
- "Client decides the response shape — that's the whole point."
- "Schema is the menu, resolvers are the kitchen."
- "N+1 is the classic trap; DataLoader batching is the classic fix."
- "GraphQL trades easy HTTP caching for query flexibility."
- "It's usually a gateway in front of microservices, not a replacement for them."

## gRPC

> Audience: a candidate with **zero hands-on gRPC**. Goal: sound informed, reason about tradeoffs, and not get caught flat if Nerdy probes it. You don't need to have shipped gRPC — you need an accurate mental model and honest framing ("I haven't used it in production, but here's how I understand it and when I'd reach for it").

### What you actually need to know (the primer)

- **What it is.** gRPC is a high-performance **RPC (Remote Procedure Call)** framework open-sourced by Google. Instead of thinking in URLs and verbs like REST, you call a **method on a remote service as if it were a local function** (`client.GetUser(req)`), and gRPC handles serialization, transport, and wiring. The "g" is just Google's framework name, not "Google-only" — it's a CNCF project used industry-wide.

- **Protocol Buffers (protobuf) / the IDL.** You define your API once in a `.proto` file — an **Interface Definition Language**. It declares `message` types (the data shapes) and `service` definitions (the methods, with their request/response types). Example:
  ```proto
  syntax = "proto3";
  service UserService {
    rpc GetUser (GetUserRequest) returns (User);
  }
  message GetUserRequest { string id = 1; }
  message User { string id = 1; string name = 2; }
  ```
  The `= 1`, `= 2` are **field tags** (stable wire identifiers), not values. Protobuf is a **binary**, schema-driven serialization format — much smaller and faster to encode/decode than JSON, but **not human-readable** on the wire.

- **Code generation / typed clients.** You run the protobuf compiler (`protoc`, or build-tool plugins) against the `.proto` and it **generates strongly-typed client and server stubs** in many languages (Go, Java, Python, C++, TypeScript/Node, C#, etc.). This is a major selling point: the `.proto` is a **single source of truth / contract**, and both sides get type-safe code for free. No hand-writing request/response models or guessing JSON field names.

- **Runs on HTTP/2.** gRPC uses **HTTP/2** as its transport, and that's where a lot of its performance and feature set come from: a **single long-lived connection** carries many concurrent requests (**multiplexing**, no head-of-line blocking at the HTTP/1.1 level), **binary framing**, **header compression (HPACK)**, and **bidirectional streaming**. Contrast with typical REST over HTTP/1.1, which is one request/response per connection turn.

- **Four call types.** This is a favorite interview detail:
  1. **Unary** — one request, one response (the "normal" RPC, closest to REST).
  2. **Server streaming** — one request, a **stream** of responses (e.g. "subscribe to a feed", live progress, large result sets).
  3. **Client streaming** — a **stream** of requests, one response (e.g. uploading chunks, then a summary).
  4. **Bidirectional streaming** — both sides stream independently over the same connection (e.g. real-time chat, live collaboration, telemetry). All built on HTTP/2 streams.

- **How it differs from REST.** REST is an architectural style over HTTP using resources/URLs + verbs (GET/POST/...) and usually JSON; it's **human-readable, browser-native, cacheable, and loosely coupled**. gRPC is a **contract-first binary RPC**: faster and more compact, with **first-class streaming**, **strict typed contracts**, and **multi-language codegen** — at the cost of being **opaque on the wire**, **not directly callable from a browser** (without a proxy), and **less ad-hoc/curl-friendly**.

- **Performance characteristics.** Generally **lower latency and higher throughput** than JSON/REST because: binary protobuf is compact and cheap to (de)serialize, HTTP/2 multiplexes and compresses headers, and connections are reused. The wins are most visible in **high-volume, low-latency, service-to-service** traffic — not in a one-off call where network round-trip dominates. (Always frame perf claims as "depends on payload size, language, and load" — don't overclaim.)

- **Where a company like Nerdy would actually use it.** The honest, high-signal answer: **internal service-to-service communication** inside the backend — e.g. a scheduling/matching service talking to a user service, a billing service, a sessions/streaming service — where you control both ends, want typed contracts across teams/languages, and care about latency and throughput. It is **not** a natural fit for the **browser-facing / public API**, because browsers can't speak raw gRPC. For browser or third-party clients you'd expose **REST/JSON or GraphQL** at the edge (or use **gRPC-Web** through a proxy like Envoy). So a realistic pattern: **REST/GraphQL at the edge, gRPC between microservices internally.**

### Graduated interview questions + model answers

**1. (Basic) What is gRPC, and what's the role of Protocol Buffers?**
*Model answer / talking points:* "gRPC is a high-performance RPC framework — you call a method on a remote service as if it were local, and the framework handles transport and serialization. You define the API in a `.proto` file using Protocol Buffers, which is both the **interface definition language** and a **binary serialization format**. From that one `.proto`, a compiler generates typed client and server code in many languages, so the schema is a shared contract. The data on the wire is compact binary protobuf rather than JSON." *(Shows the core mental model: contract-first, codegen, binary.)*

**2. (Basic→mechanism) Why does gRPC use HTTP/2, and what does that give you?**
*Model answer / talking points:* "HTTP/2 gives gRPC a few things HTTP/1.1 doesn't: **multiplexing** many requests over one long-lived connection without head-of-line blocking, **binary framing**, **header compression**, and **streaming in both directions**. That's what makes gRPC efficient for chatty service-to-service traffic and what enables its streaming call types. Reusing one connection also avoids repeated connection setup overhead." *(If pushed: HTTP/2's stream concept is what the four call types are built on.)*

**3. (Knowledge depth) Walk me through the four kinds of gRPC calls. When would you use streaming?**
*Model answer / talking points:* "Unary is one-request/one-response — the default, like a normal REST call. Server streaming is one request and a stream of responses — good for subscriptions, live progress, or paging through a big result set. Client streaming is a stream of requests and one response — good for uploading chunks and getting a summary back. Bidirectional streaming is both sides streaming at once over the same connection — good for real-time things like chat, live collaboration, or continuous telemetry. I'd reach for streaming when data is **continuous or large** and I don't want to make the caller poll repeatedly." *(Concrete examples are what sell this.)*

**4. (Tradeoffs) gRPC vs REST — what are the real tradeoffs, and what do you give up by choosing gRPC?**
*Model answer / talking points:* "gRPC's wins: it's **faster and more compact** (binary protobuf + HTTP/2), gives you **typed contracts and codegen** across languages, and has **first-class streaming**. What you give up: it's **not human-readable** on the wire, so it's harder to debug with curl or a browser; it's **not directly callable from browsers** without gRPC-Web and a proxy; HTTP-level **caching and the broad tooling/ecosystem** around REST/JSON are weaker; and you take on the **build-time codegen step** and schema management. REST is simpler, more universal, and easier for ad-hoc/public consumption. So it's: **performance + strict contracts vs. ubiquity + simplicity.**" *(Naming what you give up signals maturity.)*

**5. (Judgment) When would you NOT use gRPC?**
*Model answer / talking points:* "I'd avoid it for **public or browser-facing APIs**, because browsers and third-party developers expect REST/JSON (or GraphQL) and can't speak raw gRPC. I'd avoid it where **human-readability and easy debugging/curl-ability** matter more than raw speed, or for **simple, low-volume CRUD** where REST's simplicity wins and gRPC's perf advantage is negligible. I'd also think twice if the **team has no gRPC/protobuf experience and no need for it** — the tooling, proxies (e.g. Envoy for gRPC-Web), and schema discipline add operational overhead that has to be justified. And where **HTTP caching at the edge** is a big lever, REST fits better." *(The "why not" is the question Nerdy is most likely to use to separate buzzword from understanding.)*

**6. (Applied / company-specific) Where in a platform like Nerdy's would gRPC fit, and where wouldn't it?**
*Model answer / talking points:* "I'd expect gRPC **internally, service-to-service** — say between a tutor-matching/scheduling service, a user/identity service, a billing service, and a live-session service — where Nerdy controls both ends, wants **typed contracts shared across teams and languages**, and cares about latency and throughput. Streaming could fit **live tutoring/session** features (real-time events, presence, telemetry). I would **not** put raw gRPC on the **browser/public edge** — I'd keep REST/GraphQL there for the web and mobile clients, optionally bridging with **gRPC-Web** through a proxy. So the shape is: **REST/GraphQL at the edge, gRPC between microservices.** I want to be upfront that I haven't run gRPC in production myself, but that's the architecture I'd reason from." *(Honest framing + a concrete, plausible architecture is the strongest possible answer for a zero-experience candidate.)*

### Two terms worth being able to define if asked
- **gRPC-Web:** a variant + proxy layer (commonly via Envoy) that lets **browser** JavaScript talk to gRPC services, since browsers can't use raw gRPC/HTTP-2 framing directly. This is the standard answer to "but how does the frontend call gRPC?"
- **Field tags / `proto3`:** the numbers in a `.proto` (`name = 2;`) are **wire field identifiers** that must stay stable for backward compatibility; `proto3` is the current protobuf syntax version. Knowing this signals you understand **schema evolution** (you add fields with new tags; you don't reuse or renumber old ones).

### One-sentence honesty framing to keep ready
"I haven't used gRPC hands-on in production, but I understand it as a **contract-first, binary RPC framework over HTTP/2** that shines for **internal, high-throughput, typed service-to-service** communication — and I'd keep REST/GraphQL at the public/browser edge." *(Confident, accurate, and doesn't overclaim experience.)*
